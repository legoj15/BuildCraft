###### Changes since 26.1 Beta release 5:
- Cleanup (Steps 3+4 of the legacy-namespace plan): deleted two unused legacy script files — `data/buildcraftcore/compat/buildcraft/book.txt` and `data/buildcraftlib/compat/buildcraft/book.txt`. They were registering separate guide books (`buildcraftcore:main` and `buildcraftlib:meta`) under dead namespaces; nothing in Java references either id, no `guide.txt` populates them with entries, and `ScriptableRegistry` walks every `data/<ns>/` subfolder so they were silently being added to `GuideBookRegistry` on every reload as registered-but-empty noise. (Pre-existing follow-up flagged in the audit doc: `BCCoreCreativeTabs.java:43` calls `ItemGuide.createForBook(..., "buildcraftunofficial:config")` but no script defines that book — separate from this cleanup.) Also fixed `CraftingUtil`'s `@EventBusSubscriber(modid = "buildcraftlib")` to `"buildcraftunofficial"` so its `onServerStarted` debug handler actually registers — the modid argument has to be a registered modid for FML to wire the handler to the mod's event-bus context, and `buildcraftlib` is not registered.
- Cleanup (Step 2 of the legacy-namespace plan): moved the 32 builder compat rule JSONs from `assets/buildcraftbuilders/compat/buildcraft/builders/` to `assets/buildcraftunofficial/compat/buildcraft/builders/` (`git mv` so blame/history is preserved), and reverted `RulesLoader.loadAll()` to its plain `for (modInfo : ModList.get().getMods())` loop — the `buildcraftbuilders` fallback set added two changes ago is no longer needed because the canonical iteration now finds the files at the registered namespace. `READ_DOMAINS.add("buildcraftunofficial")` then guarantees the canonical scan happens regardless of mod ordering. Verified all 55 game tests still pass: torch/wall/leaves/fence rule-driven placement, fluid handling mode, support-required, etc. all continue to read the same rule data, just from the renamed path.
- Audit + cleanup: catalogued the 8 legacy `assets/` namespaces (and 2 legacy `data/` namespaces) carried over from the 1.12.2 multi-mod suite — only `buildcraftunofficial` is registered, so anything reached via `ModList.get().getMods()` iteration silently misses these. Wrote up the inventory, the active-vs-orphan classification, and a phased migration plan in [docs/legacy-namespace-audit.md](docs/legacy-namespace-audit.md). Found one more silent-bug case in the same shape as `RulesLoader`: `CraftingUtil` is annotated `@EventBusSubscriber(modid = "buildcraftlib")` and so its server-started subscriber never registers (low impact — the handler is just a debug recipe-class dump). Executed the obviously-safe deletes from Step 1 of the plan: `assets/buildcraft/logo.png` (duplicate of the active root `logo.png`), `assets/buildcraftlib/atlases/blocks.json` + `textures/block/white.png` (orphan atlas referencing only itself), `assets/buildcraftcore/recipes/list.json.old.txt` (explicit `.old`), and `assets/buildcrafttransport/recipes/_constants.json` + `_factories.json` (1.12.2 forge-recipe metadata; `_factories.json` references `buildcraft.transport.recipe.EngineRegisteredFactory`, a class that no longer exists). Larger moves (RulesLoader compat tree, lib texture references, guide markdown image fixes) are left as separate commits per the plan.
- Fix: `RulesLoader.loadAll()` now also scans the legacy `buildcraftbuilders` resource namespace, not just the registered modid namespaces. The vanilla compat rule files (`assets/buildcraftbuilders/compat/buildcraft/builders/*.json` — torches, walls, leaves, doors, fluid handling, fence/pane connection ignores, redstone-cycling carve-outs, etc.) live under that namespace as a holdover from 1.12.2 when BuildCraft was a multi-mod suite (`buildcraftcore`, `buildcraftbuilders`, …). The 26.1.x port merged the source into a single `buildcraftunofficial` mod but left the resource files at their historical paths; since `ModList.get()` never yields a mod named `buildcraftbuilders`, the entire rule set was silently failing to load — every block ended up with empty `requiredBlockOffsets`/`ignoredProperties`/`placeBlock` and the Builder lost dependency-aware placement ordering plus the "ignore neighbour-cycling properties" carve-outs. (This also explains why every previous "modernized the JSON rules" entry above this one was a no-op until now — those files were correct, just never read. The torch fix landed earlier still works because it ran on the canSurvive side, independent of rules.) Adding `buildcraftbuilders` to the scanned namespace list in `loadAll()` is the minimal fix, leaving everyone else's blueprint resource paths untouched.
- Fix: `SchematicBlockDefault.deserializeNBT` now re-derives `ignoredProperties` from the current JSON rules instead of trusting only the saved values. Old blueprints (scanned before a rule was added — e.g. before `multiple_variants.json` got the wall connection / leaves persistent carve-outs) had empty `ignoredProperties` baked into their NBT, and `init()` is only called at scan time, not at load time, so without this migration the user would have to re-scan every blueprint after every rule update. The migration is additive (saved entries are kept, new ones from rules are added by name) so user-customised rule files won't have entries silently dropped. Two new game tests `buildcraftunofficial:builder_deserialize_migrates_{wall,leaves}_ignored_properties` cover the load path.
- Fix: Builder no longer loops break→place forever on leaves and on neighbour-aware blocks (cobblestone walls and the rest of the WallBlock family, every wood/nether-brick fence, iron bars, glass panes, all 16 stained glass panes). Two distinct loops:
  - Leaves: schematics captured natural foliage with `persistent=false`. Once the Builder placed them away from a log, vanilla's randomTick pushed `distance` to 7, `decaying()` fired, and the leaves dropped to air. `isBuilt` then said world≠schematic and re-queued the position; the next placement decayed again. `SchematicBlockDefault.build()` now forces `PERSISTENT=true` on any LeavesBlock state right before `setBlock` (mirrors what vanilla `LeavesBlock.getStateForPlacement` does for player-placed leaves).
  - Connection-aware blocks: the schematic captured the `north/east/south/west`(/`up` for walls) connection state from the source world; after Builder placement, vanilla `updateShape` rewrote those values to match the new neighbours. `isBuilt`'s strict equality saw the diff and marked the position TO_BREAK. Added `multiple_variants.json` entries that put `north/east/south/west(/up)` into `ignoredProperties` for all walls, fences, iron bars, and glass panes — same approach as the existing `lit`/`powered` carve-outs for redstone state cycling. Connections still end up correct in-world because `updateShape` keeps maintaining them; we just stop re-comparing them at `isBuilt` time. Redstone wire is intentionally NOT included — its connection enum encodes player intent (visual run direction) so the strict-equality semantics remain correct.
  - Three new game tests `buildcraftunofficial:builder_leaves_placed_persistent`, `builder_wall_isbuilt_ignores_connections`, `builder_fence_isbuilt_ignores_connections` cover both fixes.
- Modernized the Builder's vanilla compat rule files (`assets/buildcraftbuilders/compat/buildcraft/builders/vanilla/{facings,block_below_required,multiple_variants}.json`) so dependency-aware placement ordering finally applies to torches, signs, banners, doors, levers, flowers, saplings, beds, rails, hanging-from-wall blocks, and the redstone-state cycling blocks. The legacy files used 1.12.2 flat-block IDs (e.g. `minecraft:torch[facing=north]`, `minecraft:wooden_door`, `minecraft:tallgrass`, `minecraft:reeds`, `minecraft:wall_sign`, `minecraft:standing_banner`, `minecraft:powered_repeater` …) that don't match modern split blocks (`wall_torch`, per-wood doors, `short_grass`/`fern`/`tall_grass`, `sugar_cane`, per-wood `*_wall_sign`, per-color `*_wall_banner`, `repeater[powered]`, `redstone_torch[lit]`, `furnace[lit]`, …), so `RulesLoader` had been silently producing empty rule sets for all of them. With proper `requiredBlockOffsets`, `BlueprintBuilder.isReadyToPlace` now defers a torch/sign/door/etc. until its support is built, eliminating the wasted animation cycle the canSurvive guard had to absorb. The `multiple_variants.json` entries also pivoted from the legacy "two block IDs that share a place item" pattern to modern "one block ID, ignore the cycling property" — preventing redstone torches, repeaters, comparators, redstone lamps, redstone ore, furnaces, blast furnaces, smokers, and campfires from re-entering break+place loops every time their `lit`/`powered` state cycles.
- Fix: Builder no longer silently consumes torches, ladders, signs, levers, redstone wire, doors, crops, carpets — any block whose `canSurvive()` requires a neighbouring support — when those supports aren't yet in place. Vanilla `Level.setBlock` doesn't validate `canSurvive`, so an unsupported block was set in the world for a tick, popped off by the immediate neighbour-update tick, but `SchematicBlockDefault.build()` had already returned true so `BlueprintBuilder.cancelPlaceTask` never refunded the item. `build()` now rejects placements whose final blockstate (after the REPLACE/CLEAR waterlog toggle, if any) can't survive at the target position; the items are refunded and the position is retried automatically once its support is built. New game tests `buildcraftunofficial:builder_support_*` cover floor torches, wall torches with/without support, ladders without a wall, the no-regression case for unanchored fences, and the sea-pickle-over-water canSurvive-after-waterlog ordering.
- Builder: added a fluid-handling mode button (top-right of the GUI, above the resource grid). Cycles through three states on click — "Do not replace fluids" (🚫, the legacy skip-fluid behavior and new default), "Replace fluids with placed blocks" (bricks icon, waterlogs the placed block if it supports vanilla `WATERLOGGED` otherwise destroys the fluid source), and "Clear out ALL fluid sources" (bucket icon, same as Replace plus destroys fluid sources at schematic-air positions so entire ponds/lava lakes drain). Blueprint-specified fluids are preserved under Clear so water-feature blueprints don't oscillate. Mode persists via NBT, syncs over ContainerData, and invalidates check results on change so flip-flipping a live Builder picks up the new behavior on the next tick. The button uses the vanilla Minecraft `widget/button` 9-sliced sprite via `AbstractButton.extractDefaultSprite`, so its bevel + hover state match the Replacer GUI's Replace button (and any other vanilla Button in the mod) and respect resource-pack overrides for free. New game tests `buildcraftunofficial:fluidmode_*` cover the waterlog/destroy branches, the lava-isn't-water edge case, flowing-vs-source handling, and the enum cycle/ordinal-clamp invariants.
- Fix: Builder no longer break/places waterlogged blocks every check tick (wasting items and power). Under REPLACE/CLEAR the freshly placed block ends up with `WATERLOGGED=true` over a water source, but the architect captured it dry, so `SchematicBlockDefault.isBuilt` saw `world.WATERLOGGED=true ≠ schematic.WATERLOGGED=false` and re-marked the position TO_BREAK forever. `isBuilt` now treats world-wet vs schematic-dry as built (not the reverse — a genuinely waterlogged schematic still demands a waterlogged world). Two new `fluidmode_isbuilt_*` regression tests cover both directions.
- Fix: Blueprint 3D preview (anywhere it appears — blueprint/schematic tooltips, Architect GUI, Replacer GUI) now rotates around the structure's true center. Vanilla's `ItemTransform.NO_TRANSFORM.apply` implicitly translates items by `(-0.5, -0.5, -0.5)` so submitted items render centered on the pose origin; the PiP renderer was translating each cell by `(x, y, z)` on the assumption that items render from their BSW corner. The resulting half-cell offset was invisible in larger blueprints but made single-block schematic previews rotate around a corner. Per-cell translate is now `(x + 0.5, y + 0.5, z + 0.5)` to compensate.

Single-block schematics are no longer consumed — the same from/to pair can be reused on a different blueprint. `Blueprint.replace` now matches on `blockState + placeBlock` rather than full object equality, plus adds `countMatchingPaletteEntries` / `countMatchingCells` helpers for the summary. Single-block schematic items now show the captured block's name in their tooltip and get a 3D preview panel below the main tooltip (via `SchematicSingleTooltipOverlay`, parallel to the existing blueprint tooltip handler). Added game tests under `buildcraftunofficial:blueprint_replace_*` covering the scan-context match regression, multi-occurrence replace, data[] invariance, no-match, counting, and null-safe no-op.
- Electronic Library GUI polish: the progress indicators now reveal the pre-baked filled-arrow sprites on the GUI texture sheet (at UV 234,224 for → and 234,240 for ←) instead of drawing coloured rectangles, so you see an actual arrow "lighting up" rather than a bar — top row (←) reveals right→left, bottom row (→) reveals left→right, matching each arrow's pointing direction. - Fix: Fixed a server crash in `BlueprintBuilder.cancelPlaceTask` when a fluid place task was cancelled after the fluid-placement deadlock fix enabled them to actually queue. The extract path's "pack fluid info into the resulting bucket item's CustomData" step had been TODO-stubbed to write an empty `BuilderFluidStack` tag, and the corresponding cancel-path `map(stack -> return FluidStack.EMPTY)` then fed `FluidResource.EMPTY` into `tankManager.insert`, tripping NeoForge's `checkNonEmptyNonNegative` precondition with `IllegalArgumentException: Expected resource to be non-empty: minecraft:empty [0]`. Now the extract path actually serialises the fluid id + amount into the bucket's CustomData, and cancel reads it back out to reconstitute the FluidStack for the tank refund — with a defensive filter so any empty/unresolvable fluid (old save data, unregistered fluid) is skipped instead of crashing.
- Fix: Builder now actually places source fluid blocks (water, lava, oil, fuel, …) that appear in a blueprint. Fluid-vs-solid constraints (wait-for-floor, wait-for-walls) are preserved.

Fix: Fixed an issue where filling a stalled Filler's inventory would not cause it to immediately resume. Inventory interactions now correctly clear the builder's required resources cache and unset the completion flag.
Fix: Fixed a visual bug where the Filler's internal inventory count would flicker and ghost items would appear in the GUI. The root cause was the client-side network sync (every 5 ticks) calling cancel() which refunded reserved items back into the visible inventory, and then the Container sync immediately correcting them. Fixed by guarding the item refund to server-only, skipping redundant buildingInfo rebuilds on the client, and stripping inventory data from the block entity update packet.
- Fix: Restored legacy 1.12.2 block breaking particles and sound effects for the Filler (and other Builder logic) by migrating from world.removeBlock back to world.destroyBlock.
- Fix: Fixed a bug where players could not top up existing item stacks in the Filler's resource inventory via the GUI. The root cause was SlotBase.getMaxStackSize(ItemStack) returning remaining insertion capacity instead of total slot capacity, causing vanilla's container logic to reject items when the slot was partially filled. This affected all BuildCraft containers using SlotBase.
- Test: Added bulk insertion gameTests to verify 4 gears can be distributed across FE Engine and MJ Dynamo upgrade slots automatically.
- Fix: Enforced maximum stack size of 1 on the FE Engine and MJ Dynamo GUI upgrade slots, preventing mouse dropping or shift-clicking an entire stack of gears into a single slot.
- Fix: Fixed a bug where the Filler would never stop operating after completing its task (e.g. clearing a volume), continuously destroying any blocks placed within the area. The missing isFinished() guard from a 1.12.2 TODO has been restored, so the Filler now correctly halts when mode is ON and the pattern is complete, while still looping when mode is LOOP.
- Fix: Fixed a bug where the Filler would falsely report itself as finished (is_finished = true) before it had actually scanned and processed the entire volume. The SnapshotBuilder only checks 10 blocks per tick, but the isDone logic was ignoring unchecked (UNKNOWN) blocks and in-progress tasks, causing it to prematurely declare completion after just a handful of blocks were processed.
- Fix: Fixed a legacy 1.12.2 bug where Pipe Gate actions that set the Filler's pattern (e.g., "Filler Action: Box") would lock the GUI visually, but failed to actually update the Filler's physical logic or prevent the player from forcefully overwriting the task in the interface. The GUI now strictly prevents interacting with the Pattern Statement slot while locked, the Server actively rejects forced manipulation packets, and the Builder correctly initiates the gate-assigned task upon receiving it.
- Fix: Fixed the Gate connector button (the vertical link between logic slots on Iron AND and higher gates) being a no-op. AbstractContainerScreen.mouseClicked() was swallowing the click before the connector hit-test could run. Reordered mouse handling to check connector buttons first and added optimistic client-side state update for instant visual feedback. Also fixed initial connection state sync when the Gate GUI is opened.

- Replaced deprecated AbstractContainerScreen GUI getter methods with their modern getters to fix compilation removal warnings.
- Enhancement: Ported the contextual Help Ledger text for the Filler block (Excavate, Invert, Mode, Locked) from 1.12.2.
- Fix: Restored the 1.12.2 Filler inventory filter that only accepts block-type items. Previously, any item (swords, sticks, ingots, etc.) could be placed into the Filler's resource slots, causing it to attempt building with unbuildable items. The filter now checks for BlockItem instances, matching the original ItemBlocks.getList() behavior.
- Test: Added FillerInventoryTester game test to verify both block items are accepted and non-block items are rejected by the Filler's inventory.
- Fix: Fixed a missing rendering implementation for the Architect Table where the "blue caution lasers" indicating the selected building volume bounds were not rendering on the client when placed next to volume markers. State synchronization and block entity lifecycle registration have been implemented, bringing back the volume visualizations.
