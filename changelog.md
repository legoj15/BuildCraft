###### Changes since 26.1.x Beta release 6:
- Restore the 1.12.2 Heat Exchanger fluid restrictions. The port lost both `tankInput.setFilter(isHeatant/isCoolant)` and `tankOutput.setCanFill(false)`, so any fluid (lava, water, oil, ...) could be inserted into any tank of the multi-block via a side-touching pipe, a bucket right-click, or a click in the GUI's tank widget — making the START accept coolants, the END accept heatants, and both output tanks accept arbitrary external fluids. The START's `tankInput` now overrides `isValid` to accept only fluids registered with `IRefineryRecipeManager.getHeatableRegistry()` (water, oil, fuel, ...) and the END's `tankInput` only fluids in `getCoolableRegistry()` (lava, hot oil, hot fuel, ...). Both output tanks are now instances of a new `OutputTank` subclass whose `isValid` returns false to externals; the craft loop fills them via `insertInternal`, which flips a guard for the duration of one `super.insert` call so recipe results still land. NBT load uses `set()` directly to bypass both the heatant/coolant filter and the output gate (saved fluid is trusted, and the recipe registry may not be ready during early chunk load). New `HeatExchangerTester` covers the OutputTank gate in three game-tests: external insert returns 0, internal insert succeeds for the same fluid, and the internal-insert flag resets after the call so a follow-up external insert is still rejected.
- Wire up the Heat Exchanger GUI. The `heat_exchanger.png` asset has been shipped since the port but no `Container`/`Screen`/`MenuType` ever pointed at it, so right-clicking only ever did bucket transfer. `TileHeatExchange` now implements `MenuProvider`, `BlockHeatExchange` opens the menu after a failed bucket interaction (matching the Distiller pattern), and right-clicking any block in the multi-block walks clockwise to find the START tile via the new `findStart()` helper so the menu always points at the section that owns the recipe state and the `endSection` link. The screen lays out four `GuiElementFluidTank` widgets in the 1.12.2 positions — END.tankInput as the top-left vertical (1.12.2 inCoolable, hot fluid), START.tankInput as the mid-left horizontal (inHeatable, cold fluid), END.tankOutput as the top-right horizontal (outHeated), START.tankOutput as the mid-right vertical (outCooled) — each with a tank-shape gauge overlay sourced from the spare `(0,171) 16x38` and `(17,171) 34x17` regions of the texture, and hover tooltips showing fluid name and `mB / capacity`; tanks whose section isn't formed yet are skipped so an isolated or partially-built exchanger still opens cleanly. The four 1.12.2 bucket / fluid-shard container slots are wired up as a `(4, 1)` `ItemHandlerSimple` on `TileHeatExchange`: every 5 ticks the START's `processContainerSlots()` drains filled containers in slots 0/1 into END.tankInput / START.tankInput respectively and fills empty containers in slots 2/3 from END.tankOutput / START.tankOutput, with `quickMoveStack` shift-click ordering between the four slots and the player inventory; slot contents persist via NBT and drop on block break. The center processing region between the tanks gets a new wipe-in/wipe-out overlay sourced from `(176,71) 54x71`: as `ExchangeSectionStart.progressState` advances OFF→PREPARING→RUNNING→STOPPING→OFF, the overlay wipes in from the left during PREPARING (visible width grows with `getProgress(partialTicks)`), holds at full 54×71 during RUNNING, then clears from the left during STOPPING (the visible region's left edge marches rightward) — visualizing the two fluids meeting in the middle and then dispersing. The vanilla title and inventory labels are suppressed because the layout is too cramped to fit either without overlapping the pipe schematic or the slot rows. Reuses the existing `block.buildcraftunofficial.heat_exchange` lang key for the menu title.
- Restore 1.12.2's coloured colour-name labels: the dye colour name in tooltips and gate descriptions (e.g. "Red" in "Red Extraction Preset" on the Emzuli pipe's gate, the colour names on paint actions and pipe-signal triggers, the coloured prefix on a paintbrush/wire/painted-pipe item name, and the lens/filter colour name) now renders in its matching `ChatFormatting` colour. `ColourUtil.getTextFullTooltip(DyeColor)` was returning plain title-cased text post-port; now it wraps the name in the appropriate colour code (using `COLOUR_TO_FORMAT`, which already remaps black to `DARK_GRAY` for tooltip visibility) and a trailing reset, mirroring the existing `Direction` overload. Removed the redundant manual `.withStyle(convertColourToTextFormat(...))` calls at the four call sites that were compensating for the missing formatting (paintbrush, pipe holder, wire, emzuli paint button tooltip), so all colour-label callers — including the gate-action and gate-trigger descriptions that hadn't been compensating — pick up the colouring through a single source.
- Move the entire visible logic-gate body from chunk-mesh-baked geometry to per-frame BlockEntityRenderer rendering, eliminating the client-side FPS spikes (~6 ms → 80–95 ms) that occurred whenever a gate changed state — every gate ON/OFF transition (e.g. a timer-driven gate firing every 10 ticks), every redstone toggle on a driven block (the lamp's `setBlock(UPDATE_ALL)` cascading back through the pipe's `neighborChanged`), and every block placement within the 27-section dirty cube around a pipe network. The chunk-mesh path put `logic.isOn` and the gate's static parts (base material brick, logic icon, modifier dots) into the chunk geometry hash, so each transition invalidated the cached pipe geometry and forced a re-mesh whose per-vertex AO + lighting cost scaled linearly with the gate count nearby. The new `PlugGateRenderer` (using the existing `IPlugDynamicRenderer` plumbing the pulsar already had) renders all gate geometry per-frame from the live `PluggableGate.logic.isOn`, `PlugGateBaker.bake` returns an empty quad list, and `KeyPlugGate.equals/hashCode` exclude `active` so the geometry hash is stable across ON/OFF. Chunks with N gates re-mesh at the same speed as chunks with 0 gates; gates render at proper world brightness with vanilla face shading by sampling lightmap from `pipePos.relative(plug.side)` and keeping `createFace`'s correct outward normals. Several smaller fixes ride along: `Pipe.updateConnections` only schedules its network update when `connected` actually changes (cuts a feedback loop where the lamp's UPDATE_NEIGHBORS bounced back through the pipe and forced a full BE sync every redstone cycle), `TilePipeHolder.replacePluggable` rebuilds the wire system when an `IWireEmitter` pluggable is added or removed (long-standing bug where placing a gate after wires already existed meant the wires never glowed because the wire system was built without that emitter element), and the four `Block.UPDATE_ALL` → `Block.UPDATE_CLIENTS` calls on the pipe-holder render-sync paths drop a duplicate server-side `updateNeighborsAt` cascade. Pulsar pulse-state transitions ride a 3-byte custom payload via `IPipeHolder.sendMessage` instead of forcing a full BE sync via `scheduleNetworkUpdate`.
- Bump NeoForge to `26.1.2.30-beta` (was `.17`). Pulls in mod-loading restructure, `ValueInput#keySet` / `FlatLevelSource` / `ComposedModelState` / `TagEmptyCondition` fixes, fail-fast on configs whose defaults don't accept their own validators, and refreshed Sponge Mixin / FancyModLoader / NeoForge Bus dependencies. None of the breaking changes in the range (GLM priority system, `BreakBlockEvent` split, `IBaseRailBlockExtension` hook removal) touch this codebase.
- Restore the six "face the &lt;direction&gt; side" gate actions for iron, wood, and daizuli pipes; the action icons were missing from the gate GUI's action picker and triggering them did nothing. Each pipe filters which directions are offered via its existing `canFaceDirection` rule (iron and wood require an active connection, daizuli accepts all six).
- Restore the 16 "Paint Items &lt;colour&gt;" gate actions on lapis and daizuli pipes. Two regressions: no pipe added them to its gate's possible-actions list (so they never appeared in the action picker), and the sprite holder pointed at `items/paintbrush/<colour>` (legacy 1.12.2 path) instead of the canonical `item/paintbrush/<colour>` so the icons rendered as the magenta missing-texture box even after the actions were re-registered. Triggering one now sets the pipe's paint colour the same way the wrench does.
- Teach `SpriteHolderRegistry.SpriteHolder.getSprite()` to walk multiple texture atlases (blocks → items → GUI) instead of hard-coding `LOCATION_BLOCKS`. Modern Minecraft (1.21+) split the historical single atlas into per-purpose atlas pages; sprites referenced by item models — like the paintbrush colour textures gate "Paint Items" actions reuse for their icons — auto-stitch into `textures/atlas/items.png` and were unreachable from the blocks-atlas-only lookup, rendering as the missing-texture box on the guide-book contents page. The fall-back resolves each PNG against its actual atlas page so each texture lives on exactly one atlas (no double-stitching, no `Duplicate sprite … will be rejected in a future version` warnings) while remaining accessible to all callers; resolution caches on first query so the cost is paid once per holder. Also dropped the redundant `single` declarations of `gui/empty_filtered_buffer_slot` and `gui/nothing_filtered_buffer_slot` from `assets/minecraft/atlases/gui.json` — both sprites are already covered by the `gui/` directory source on the blocks atlas, and the SpriteHolder fall-back finds them there without the duplicate warning.
