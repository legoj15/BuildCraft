###### Changes since 26.1 Beta release 5:
- Fix: Snow / carpet / similar fragile placements where the schematic itself has air at the support position (the architect captured a vanilla-generation glitch — snow over air, snow on partial snow_layer with non-full top, etc.) now defer at queue time instead of looping through throw → undo → refund → re-queue. The `requiredBlockOffsets` dependency check is vacuously CORRECT when the schematic at the support position is air (schematic-air + world-air → CORRECT), so the position queues even though no real support exists. Added a `canSurvive` pre-check in `BlueprintBuilder.isReadyToPlace` that defers placements which would fail the post-setBlock self-destruct check anyway — this stops the wasted animation cycles. Trade-off: torch on top of a cobblestone wall whose `up=false` (a rare configuration where vanilla's `shouldRaisePost` returns false because the wall has TALL connections on opposite axes) will now defer permanently — vanilla's normal player-placement cascade would set `up=true` upon torch placement, but the queue-time defer never lets the placement happen. If you hit this, placing the torch manually triggers vanilla's normal cascade and works.
- Fix: Old blueprints (scanned before a JSON rule modernisation) now migrate their `requiredBlockOffsets` from current rules when the saved set is empty. Symptom: a blueprint scanned back when `facings.json` still used legacy `minecraft:torch[facing=up]` selectors (which never matched the modern `minecraft:torch` block) saved torches with empty offsets in NBT; those torches' `isReadyToPlace` returned true vacuously (empty `allMatch` → `true`) and queued regardless of whether their support was built, producing a throw→undo→refund→re-queue loop the user observed as "torches floating in the sky on repeat" once the supporting walls couldn't be placed (out of resources). The migration runs in `SchematicBlockDefault.deserializeNBT` only when the saved offset set is empty — a non-empty saved set was written deliberately (potentially with rotation applied) and unconditionally adding un-rotated rule offsets on top would over-defer rotated blueprints. Old broken saves are exactly the empty-set case, so the targeted check fixes them without breaking rotated saves.
- Fix: Torches placed on top of cobblestone walls (and similar "block-below has up-property" setups) no longer cycle in throw→destroy→throw forever. The pre-setBlock `canSurvive` check in `SchematicBlockDefault.build()` evaluated against the *current* world state, which for a freshly-placed wall reads `up=false` (no neighbour above yet) — so canSurvive failed for the torch and the placement deferred forever, even though vanilla's setBlock cascade would have set the wall's `up=true` because of the torch above and the torch would have been fine. Replaced with a post-`setBlock` validation that uses the existing `updateShape` iteration: if `updateShape` returns AIR for any direction (the block self-destructed because vanilla decided it can't survive), the placement is rolled back (including the atomic other-half if it's a bed FOOT or door LOWER) and items are refunded via the existing `cancelPlaceTask` path. This reflects post-cascade reality rather than pre-cascade pessimism, so torch-on-wall and similar "cascade fixes my support" cases now work, while torches in genuinely unsupported positions still get caught (vanilla `updateShape` returns AIR, my iteration undoes). The atomic dual-half placement for beds/doors had to move BEFORE the updateShape iteration so the iteration's other-half-direction call doesn't mistake the missing other half as a self-destruct trigger.
- Fix: Doors now consume only one door item per door (was two), with no half-door placements left in the world if either half is blocked. Same shape as the bed atomic-dual-half fix one entry below: `DoorBlock` instances are detected via instanceof in `addClassBasedRequiredBlockOffsets` (covers all 14+ vanilla wood/iron/copper/nether door variants and any modded doors automatically), the UPPER half returns empty `computeRequiredItems` (so the resource panel shows the right count), `build()` for LOWER pre-checks the UPPER position is clear/replaceable and atomically setBlocks both halves with the LOWER's FACING/HINGE/OPEN/POWERED state copied through to UPPER (only HALF differs). Torches don't need analogous treatment — they're single blocks already covered by `facings.json` (floor torches require block-below; wall torches require the wall behind them) and have been deferring correctly via `requiredBlockOffsets` since the JSON modernization a few entries up.
- Fix: Beds now consume only one bed item per bed (was two), and the place no longer leaves a half-bed in the world if either position is blocked. Vanilla beds are two adjacent half-blocks (FOOT + HEAD) sharing a single bed item; the architect captures the halves separately, which used to mean the Builder consumed two bed items per bed and produced visible half-bed placements. Now: HEAD positions return an empty required-items list (so the resource panel shows the right count, and no extraction happens) and have a `requiredBlockOffset` pointing at the FOOT position (so the HEAD never queues an independent place task — it flips to CORRECT after the FOOT places both halves). The FOOT additionally requires the block below the HEAD position so vanilla `updateShape` doesn't destroy a bed whose HEAD half lacks support, and FOOT's `build()` pre-checks that the HEAD position is empty/replaceable before running setBlock — if not, the placement defers (item refunded) until the HEAD position is cleared. After both checks pass, FOOT's build atomically setBlocks both halves. Class-based detection in `addClassBasedRequiredBlockOffsets` (called from both `setIgnoredProperties` and `deserializeNBT`) means any vanilla or modded BedBlock subclass picks up the linkage automatically — no need to enumerate all 16 colours.
- Fix: Snow layers, carpets (16 colours + moss + pale_moss), and pink_petals now have their `block-below-required` rule entry, so the Builder defers their placement until the supporting block beneath them is built. Without this, the Builder would queue a snow_layer placement at a position with no support, the per-build `canSurvive` defer would catch it (return false → `cancelPlaceTask` refunds the item), the position would re-classify as TO_PLACE next check, the cycle would repeat — wasting a place-task animation cycle and a tick of battery power per attempt until the supporting block happened to be placed. Now those placements simply don't queue until the position-below check result reads CORRECT, matching how torches / saplings / doors / etc. already behave.
- Fix: Builder-placed glass panes, fences, walls, redstone wire, and similar neighbour-aware blocks now actually connect to their neighbours after placement. Vanilla `setBlock` with flag 11 calls `updateShape` on each *neighbour* of the placed block (so they can update their connections to the new block) but doesn't call updateShape on the placed block itself. Combined with the `ignoredProperties` carve-out resetting connection bits to the default no-connection state at build time, that left every Builder-placed pane as a visible single column even when surrounded by valid connectible blocks; breaking and re-placing the pane by hand worked because vanilla's `getStateForPlacement` chain runs updateShape during player placement. `SchematicBlockDefault.build` now iterates each of the six neighbours after `setBlock` and applies `BlockState.updateShape`, mirroring what player placement does. Skipped if updateShape returns AIR (would mean the block decided it can't survive — canSurvive above should have caught that, and removing a freshly placed block is unsafe).
- Fix: Stair blocks (any vanilla or modded `StairBlock` instance) now have their `shape` property auto-added to `ignoredProperties` for placement comparison, regardless of which stair variant. Vanilla has 50+ stair block IDs across wood / stone / brick / nether / deepslate / copper variants; listing them all in `multiple_variants.json` is brittle and doesn't help with modded stairs. Without this carve-out, stairs near other stairs (or near a fluid event that nudged neighbours) cycled in break+replace forever — `SchematicBlockDefault.isBuilt`'s strict comparison saw the schematic-captured shape vs the world-recomputed shape (vanilla updates corner shapes inner_left/outer_right based on actual neighbours) as a mismatch even when the build was semantically complete. New `addClassBasedIgnoredProperties()` helper called from `setIgnoredProperties` (fresh scans) and `deserializeNBT` (old saved schematics get migrated automatically). Same hook can be extended for other shape-aware classes if needed.
- Fix: Builder no longer cycles fragile schematic blocks (snow_layer, carpet, button, redstone wire, sapling, torch, sign, lever — anything where `canBeReplaced(state, water)` returns true) through an extract-then-defer-then-refund loop in REPLACE mode when the build area has fluid 2+ cells away that flows in over a few ticks. The per-build fragile defer in `SchematicBlockDefault.build` only sees immediate-neighbour fluid; for water further out, the place task got queued (extracting items from inventory at queue time), the animation ran, the defer fired in build (because by then the water had flowed close enough to be a neighbour), the items got refunded by `cancelPlaceTask`, and the position re-classified for the next cycle. The user-visible symptom was the inventory slot count for snow layers (etc.) bouncing up and down in real time as the cycle ran. Now the place-task-add filter in `SnapshotBuilder.tick` runs the fragile check at queue time: REPLACE + area has any fluid + schematic block is fragile = don't queue, items stay in inventory. Solid blocks (canBeReplaced=false for water — stone, planks, etc.) are unaffected and proceed to placement normally; the per-build defer remains as a backstop. New `SnapshotBuilder.isFragileSchematicAt(BlockPos)` (default false), overridden in `BlueprintBuilder` to inspect the schematic's `BlockState.canBeReplaced(Fluids.WATER)`.
- Fix: Builder no longer deadlocks in CLEAR mode at positions where the schematic has a non-air block (e.g. snow_layer) and the world has water. Previous classification: `BlueprintBuilder.canPlace` returned true for those positions in REPLACE/CLEAR (the "fluid carve-out" so the place path could waterlog or destroy the source in one pass), routing them into TO_PLACE. The mop gate then deferred them because the area "still had fluid" — but the only fluid keeping the gate closed was the fluid AT THOSE POSITIONS, which couldn't get cleared because they were TO_PLACE not TO_BREAK and the break path doesn't touch TO_PLACE. Symptom: builder ignores the water in front of it forever even after the user sealed the leak. Fix: in CLEAR mode `canPlace` returns false for world-fluid positions, routing them into TO_BREAK. The mop sweeps the fluid; the position re-classifies as TO_PLACE on the next check pass once the world position is empty; the place runs after the rest of the area is also dry. REPLACE mode keeps the existing fluid-carve-out behaviour (place runs immediately and the build path waterlogs / destroys the source in the same pass — that's the cheaper one-pass option REPLACE was designed for, and it doesn't deadlock because REPLACE never closes the mop gate).
- CLEAR fluid-handling-mode spec change: blueprint-specified fluid positions are no longer preserved. The original CLEAR spec ("clear schematic-air-positioned fluids only, preserve blueprint fluids to avoid oscillating on water-feature blueprints") didn't match the user-stated intent of "clear all fluids in the build area" — a blueprint that captured water around solid features (snow trail through a stream, flooded ruin, etc.) would leave every captured-fluid position untouched in CLEAR, defeating the mode entirely. New behaviour: in CLEAR mode `BlueprintBuilder.isAir` treats `SchematicBlockFluid` instances as air for classification purposes, so a position whose blueprint specifies water becomes TO_BREAK when the world has water there and CORRECT when the world is empty (matching the air-treated schematic). Water-feature blueprints can use REPLACE if they want to keep their fluid; CLEAR is now unambiguously "I want this area dry no matter what." Updated the EnumFluidHandlingMode docstring with the new spec.
- Fix: Builder's CLEAR-mode "wait for mop to finish before placing" gate no longer deadlocks when the build area contains waterlogged source-emitting blocks (waterlogged stairs, fences, slabs, etc.). The deadlock: a waterlogged block emits water to its neighbours, so corner positions perpetually re-flood and the Builder's break beams fire forever; meanwhile the place tasks that would dry the source-emitting blocks (and stop the emission) sit deferred behind the gate because "the area still has fluid." Carve-out: waterlog-clear-only place tasks bypass the gate. Drying them is itself part of mopping (turning off the source emission) so they belong in the same phase as the fluid-break tasks, not the post-mop place phase. Implementation: new `SnapshotBuilder.isAllowedDuringFluidMop(BlockPos)` (default false), overridden in `BlueprintBuilder` to return true for positions where the schematic's `isWaterlogClearOnly` matches (CLEAR mode + same block + world wet / schematic dry). The place-task-add filter passes the carve-out through. Other placements (real block placements, fragile blocks at risk of timing-race floodout) still wait for the area to be fluid-free.
- Fix: Builder's CLEAR fluid-handling mode now actually clears water from waterlogged blocks where the schematic captured the block dry. Previously CLEAR matched REPLACE on the place path — both opportunistically waterlogged a placed block over a water source, and `SchematicBlockDefault.isBuilt` had a leniency clause that treated "world wet, schematic dry" as already-built (the original purpose was to prevent REPLACE-mode placements from looping break+place when REPLACE itself produced the world-wet state). The combination meant a wet fence whose schematic counterpart was dry was permanently considered built and the user-stated CLEAR intent ("clear all fluids") didn't apply to waterlog-style fluid. New behaviour: REPLACE keeps the opportunistic waterlog (preserve water "for free"); CLEAR respects the schematic's WATERLOGGED value strictly. Implementation: `SchematicBlockDefault.isBuilt` got a fluidMode-aware overload that disables the leniency under CLEAR, `BlueprintBuilder.isBlockCorrect` now passes the mode through, and `SchematicBlockDefault.build` differentiates the two modes when deciding whether to set WATERLOGGED on the placed state. The architect's table already captures WATERLOGGED (it's just a property on the BlockState round-tripped through `NbtUtils.writeBlockState`), so nothing changes on the scan side. To avoid wasting items when "drying" a block whose only difference from the schematic is its WATERLOGGED property, the place path detects this case (`SchematicBlockDefault.isWaterlogClearOnly`) and skips item extraction entirely — the position gets re-set with WATERLOGGED=false, no fence is consumed from inventory. Six new game tests under `buildcraftunofficial:fluidmode_clear_dries_*` / `_replace_still_opportunistically_*` / `_isbuilt_strict_in_clear` / `_isbuilt_lenient_in_replace` / `_clear_only_under_clear` / `_clear_only_requires_matching_block` cover both directions of the mode split + the helper's predicate.
- Fix: Builder break-task ordering in CLEAR mode now goes source-fluid first, then flowing-fluid, then non-fluid blocks (within each tier the existing distance-from-centre order is preserved). For mixed source+flowing fluid bodies, breaking sources first lets the remaining flowing drain naturally, which (a) cuts the total break count needed for self-contained pools, and (b) handles MC's infinite-water regeneration mechanic better — when a 2×2 pool of sources gets multiple sources broken in the same tick (now likely with the 1 MJ fluid cost from a few changes up), no remaining source has 2+ source-neighbours so the empty cell can't re-source. Doesn't help leaky-boundary cases (sources outside the build area), but the prior "infinite mop" CLEAR-mode behaviour remains the right answer there: the activity is visible so the user can extend the build area / fix the source / switch to REPLACE.
- Fix: Builder/Quarry/Filler fluid break cost is no longer absurd. Vanilla pure-fluid blocks (`Blocks.WATER`, `Blocks.LAVA`, modded LiquidBlock instances — BC oil, fuel) ship with `strength(100.0F)` — that's their *explosion resistance*, not break time, but `BlockUtil.computeBlockBreakPower`'s `(hardness+1)*2 * 16 MJ` formula reads it as time-to-break and produced 3232 MJ per fluid block (40× stone, more than obsidian). At MAX_POWER_PER_TICK = 256 MJ/tick that's ~13 ticks per single water break — the user-visible "laser fires forever, water never clears" symptom that survived both the BlockUtil setBlock-AIR fix and the battery-sync fix. Now LiquidBlock instances charge a flat 1 MJ each (matching vanilla's "free to break with empty hand" intent); non-fluid blocks keep the original formula. Three new game tests `buildcraftunofficial:blockutil_{water,lava,stone}_break_cost*` cover both directions of the carve-out.
- Fix: Builder's CLEAR fluid-handling mode now defers all place tasks until every fluid in the build area has been mopped — no more "snow placed adjacent to water corner, water flows back next tick, snow drops as item entity nobody picks up, place-task re-queues, repeat" wasted-resource loop. The fragile-block defer in `SchematicBlockDefault.build()` (added a few commits up) is still there as a backstop for REPLACE mode and other narrow cases, but it can only see immediate-neighbour fluid at the moment of placement; water flowing in from one cell over within the same or next tick lands on the freshly placed block and destroys it before the defer fires next cycle. CLEAR-mode users said "logically that should be the order of operations" (clear all fluids, then build), so `SnapshotBuilder.tick`'s place-task-add gate now also requires `!buildAreaHasAnyFluid()` under CLEAR. Trade-off: at a leaky boundary (fluid source outside the build area) the Builder will mop forever and never start placing — but the activity is visible (break beams keep firing) so the user can extend the area / fix the leak / switch to REPLACE rather than wonder why nothing's happening. Added `buildAreaHasAnyFluid()` helper to `SnapshotBuilder` (O(n) box scan, short-circuits on first fluid found, only called once per tick during the place-task-add gate so the cost is negligible).
- Fix: `BlockUtil.breakBlockAndGetDrops` now actually clears pure-fluid blocks instead of silently no-op-ing them. The previous `world.destroyBlock(pos, false)` call delegates internally to `setBlock(pos, fluidState.createLegacyBlock(), 3)` — for a position whose only "block" IS the fluid (water, lava, oil, fuel — any `LiquidBlock` instance), `createLegacyBlock()` returns the same fluid state, so destroyBlock is a silent no-op while still reporting success. The Builder's CLEAR-mode break tasks were "succeeding" against water (returning `Optional.of(emptyList)`, removing the task from the queue) but the water never actually disappeared; the position re-classified as TO_BREAK on the next check, re-queued, fired the laser again, looped forever — the user-visible "phantom laser fires at water source, water never gets mopped" symptom that survived the battery-sync fix two commits up. Now LiquidBlock positions take a hard `setBlock(pos, AIR, 3)` (which clears both the BlockState and the derived FluidState); non-fluid blocks keep going through `destroyBlock` so its sound / particle / level-event side effects still fire; waterlogged blocks (block != LiquidBlock, fluid via WATERLOGGED) keep the destroyBlock + legacy-fluid-restore behaviour, which is correct there because we're breaking the block, not the fluid (the fluid clearing happens on a second break pass once the block is gone). Three new game tests under `buildcraftunofficial:blockutil_break_*` cover water source clear, flowing-water clear, and the regression case for solid blocks (stone still drops cobblestone and the position becomes air via destroyBlock). Side effect of this fix: the Builder's CLEAR-mode "perpetual mopping" at a leaky boundary will now visibly progress (each laser cycle removes the flowing fluid for ~1 fluid tick before it re-floods); previously the visual was the same but no actual work was happening. The block-throwing place animation may appear choppy while CLEAR is mopping fluid because client-side place-task extrapolation is gated on the break queue being empty — that's by-design (server prioritises break over place, and naively extrapolating place ahead of server placement would visually desync the throw landing) and resolves once the area is dry.
- Fix: Builder/Filler client/server battery sync no longer accumulates past capacity. `MjBattery.addPower(long, boolean)` is unbounded — a bare `microJoules += n` with no clamp — and the additive semantics are correct for external energy receivers (engines, MjBatteryEnergyHandler) but were also being used by `TileBuilder.loadAdditional` / `TileFiller.loadAdditional` to apply the synced battery snapshot from `ClientboundBlockEntityDataPacket`. Since that packet re-runs `loadAdditional` every 5 ticks, every sync was *adding* the server's value into the existing client value instead of replacing it. After ~13 seconds the client battery climbed to ~65× capacity (reported in the wild as 1,036,410 / 16,000 MJ on a Builder whose server-side battery sat at 0). The downstream symptom was phantom laser firing in CLEAR mode: `SnapshotBuilder.clientTick` gates animation extrapolation on `stored > 0`, so the inflated client battery let it locally charge break tasks past their target, drawing a laser at every queued source — while the server, with 0 MJ, never actually progressed any task and the water never broke. Added an absolute `MjBattery.setStored(long)` (clamps to `[0, capacity]`) and switched both load paths to it; `MjBattery.deserializeNBT` and `readFromBuffer` also route through it for consistency. The Filler's pre-existing `extractAll() + addPower()` workaround (which produced the right value but bypassed the symptom rather than fixing the cause) is replaced. Five new unit tests under `MjBatteryTest` cover the absolute-set semantics, the `[0, capacity]` clamp, the no-accumulation invariant on repeated calls, and the additive-`addPower` behaviour stays untouched. After this commit the user's "no engine, server battery 0, but client shows millions of MJ and lasers fire at nothing" scenario reads as `0/16000` on both sides, no laser; once power is connected the value rises in lockstep and the laser fires only when there's real power for a break task.
- Tuning: oil generation in oceans is now tiered, and overall ocean oil density is substantially reduced. The 26.1 default `surfaceDepositBiomes` had merged every vanilla ocean variant (shallow + deep) plus deserts/badlands into one tier with a flat 3.0× spawn-rate bonus AND eligibility for the dominant `LAKE` roll (the 6%-per-chunk surface-tendril type), giving any ocean ≈6.4% oil per chunk — about 45× the 1.12.2 vanilla-ocean rate, and the reason oil felt "guaranteed around any large body of water". `OilGenerator` now reads two separate biome lists: the existing `surfaceDepositBiomes` (light tier, 1.25× bonus, no LAKE roll) defaulting to shallow ocean variants only (`ocean`, `warm_ocean`, `lukewarm_ocean`, `cold_ocean`, `frozen_ocean` — `frozen_ocean` was missing from the prior defaults), and a new `richSurfaceDepositBiomes` (rich tier, 1.5× bonus, LAKE roll allowed) defaulting to deep ocean variants plus the existing land oil biomes (`deep_ocean`, `deep_lukewarm_ocean`, `deep_cold_ocean`, `deep_frozen_ocean`, `desert`, `badlands`, `eroded_badlands`, `wooded_badlands`). The `smallOilGenProb` default has also been quartered from 2.0% to 0.5% — this is the rate that drives the dominant LAKE-style surface tendrils, which had been flooding ocean horizons. Resulting per-chunk probabilities: plain land 0.14% (unchanged), shallow ocean ≈0.18% (1 in 572, a "slightly higher" tier), deep ocean ≈0.95% (1 in 106 — clearly the prize, but a finite-search rather than an everywhere-you-look prize), average ocean ≈0.43% (about 15× rarer than before). Behavioural change in the algorithm is one-line: the `LAKE` gate switched from `oilBiome` (light-tier-or-better) to `richBiome` (rich-tier-only), so the giant 6-block-radius surface tendrils with 25-45-block branch radii now appear only in deep oceans and on land deposit biomes rather than blanketing every coastal shallow. The `excessiveBiomes` 30× knob and `oilWellGenerationRate` global multiplier still apply on top; `mediumOilGenProb` and `largeOilGenProb` are unchanged. **Existing worlds**: TOML config files preserve their stored `smallOilGenProb` value once written, so to pick up the new 0.5% default delete `config/buildcraftunofficial-worldgen.toml` (or hand-edit the `smallOilGenProb` line) and let the mod regenerate it. New lang entry `buildcraftunofficial.configuration.richSurfaceDepositBiomes` for the config UI. The custom 1.12.2-era `BiomeOilOcean` / `BiomeOilDesert` biomes and their Simplex-noise injection layers remain unported — this tier system reproduces the "deep oceans are the oily ones" feel without re-introducing custom biomes.
- Fix: REPLACE/CLEAR fluid-handling no longer loops place→destroy→place when a "fragile" block (snow_layer, carpet, button, redstone wire, torch, sapling, sign, lever, pressure plate — anything whose `BlockBehaviour.canBeReplaced(state, fluid)` returns true) is being placed adjacent to fluid. The earlier behaviour: the REPLACE path destroyed the source at the target position and placed the block; on the next fluid tick, water from a horizontal/up neighbour (or from another source bordering the broken position) flowed back, displacing the freshly placed block; the Builder consumed one item per cycle until the inventory drained. `SchematicBlockDefault.build()` now defers placement (returns false → existing `cancelPlaceTask` refunds the item) when the resulting block is fragile (canBeReplaced returns true for the neighbour's fluid type), the block isn't being waterlogged, and any of the four horizontal neighbours OR the block above holds fluid. Solid blocks (canBeReplaced=false) and waterlogged blocks (coexist with their fluid) are exempt. Bonus structural fix: the existing source-destroy was previously fired *before* the canSurvive check, so a failing canSurvive defer would leave the world torn down with no replacement; world mutations are now deferred until every "should we abort?" check has passed.
- Fix: Builder's CLEAR fluid-handling mode now actually mops up flowing fluid, not just sources. Previous behaviour — `SnapshotBuilder.shouldBreakQueueAcceptFluid` only let sources through and assumed flowing fluid would drain naturally once every contributing source was inside the build area. That assumption fails at a leaky boundary: a build area at the edge of a pool, river, or any body whose source extends outside. There, broken positions immediately refilled with flowing fluid from outside; classified TO_BREAK, filtered out by the source-only rule, no break beam ever fired again, and the Builder appeared stuck after one or two beams. CLEAR now accepts both sources and flowing fluid through the break queue, so the Builder keeps mopping until either the area is dry (whole pool inside) or the user stops it / extends the area / deals with the external source (leaky boundary). The activity is visible — beams keep firing — so the user sees what's happening rather than wondering why nothing is. Six new game tests under `buildcraftunofficial:fluidmode_fragile_*` / `_solid_*` / `_waterloggable_*` / `_no_replace_skips_*` cover the defer's positive path, the don't-defer carve-outs (solid block, waterloggable block, NO_REPLACE mode), the UP-neighbour case, and the world-untouched invariant on defer.
- Engine power-ledger icon animations are now resource-pack overridable. `LedgerEngine.drawIcon()` and `LedgerEngineFE.drawIcon()` had been animating the active/warm engine icons in Java — manual `vOffset = (System.currentTimeMillis()/50 % 6) * 16f` against the 16×96 sprite sheet, hardcoded to 6 frames at 1 tick each. Replaced with a sprite-atlas lookup via the existing `SpriteHolderRegistry` (the engine icons live under `buildcraftunofficial:textures/icons/`, already stitched into the blocks atlas via `atlases/blocks.json`'s `icons/` directory source). `graphics.blitSprite(RenderPipelines.GUI_TEXTURED, sprite, x, y, 16, 16)` reads the atlas sprite's current-frame UVs each call, so animated entries auto-advance through Minecraft's atlas-animation ticker. The `engine_{active,warm}.png.mcmeta` sidecars finally become load-bearing — drop a resource pack with a different `frametime`, custom `frames` array, or `interpolate: true` and the change applies without touching code. `engine_inactive` and `engine_overheat` get the same code path (no animation, just static atlas sprites) so the rendering branch is uniform.
- Correction to the Step 5+6 changelog entry below: the "fix latent rendering bugs" framing for the `.mcmeta` and `buttons.png` moves was wrong. So Step 6 was a pure namespace cleanup, not a bug fix. Apologies — I'll be more careful when claiming "this was silently broken" without verifying the actual rendering path. The follow-up commit refactors `LedgerEngine`/`LedgerEngineFE` to actually use Minecraft's animation system (via the existing `SpriteHolderRegistry`/blocks atlas), which will let resource packs override `frametime`, frame order, and interpolation.
