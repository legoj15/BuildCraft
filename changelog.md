###### Changes since 26.1.x Beta release 6:
- Engine wrench UX overhaul + overheating port. **The previous behaviour** was that only crouch+RMB with a wrench rotated an engine, played the wrench sound, and granted the "Wrenched!" advancement — every other engine handled non-crouch wrench differently (Stirling/FE/Dynamo opened the GUI instead of rotating, Combustion rotated via a separate code path that bypassed the wrench's own success hook, Creative cycled power) and overheating an engine **always exploded** because the proper "sit in the overheated state" loop from 1.12.2 was never ported. **The new behaviour** unifies the engine family: non-crouch+wrench rotates (sound + advancement), crouch always opens the GUI (Redstone/Creative have no GUI), and **overheated Stirling/Combustion engines no longer explode** by default — they sit in the OVERHEAT stage producing no power until cooled, and a wrench-tap clears the overheat, plays the slide sound, and now wires up the previously-dead "Too much power!" advancement. New config option `general.canEnginesExplode` (default `false`, in the in-game NeoForge config screen) restores 1.12.2-style explosions for users who want them. Engines with no valid rotation target play the vanilla `block.lever.click` sound pitched up (1.3×) instead of the piston sound and grant no advancement, so the player gets audio + arm-swing feedback that the wrench fired but had nothing to do without trivialising the achievement. Creative engine retains its non-crouch=cycle-power behaviour but now also grants the wrench advancement on each cycle and plays the same lever click pitched *down* (0.7×) as a distinct confirmation tone; crouch+wrench rotates if possible (no power step-back when surrounded — too clever to be intuitive). Every wrench interaction that "did something meaningful" (rotation, overheat-clear, power-cycle, soft-fail no-op) now swings the arm — previously only successful rotations swung. Combustion engine's dual rotation code path is consolidated into the single base `ICustomRotationHandler` path. FE engine's crouch UX is inverted to match the rest of the family (was: crouch=rotate; now: crouch=GUI).

- "Abbreviate Large Numbers" now actually applies to MJ readouts, and fluid abbreviation shifts the unit from millibuckets to buckets instead of using k-suffixed millibuckets. **What was broken (MJ side)** — the original abbreviation design in [LocaleUtil](src/main/java/buildcraft/lib/misc/LocaleUtil.java) deliberately routed `formatDouble` *without* the abbreviation branch, so MJ-side formatters (`localizeMj`, `localizeMjFlow`) bypassed the toggle entirely. The intent at the time was to preserve precision for JEI recipe-cost labels, but it caught every MJ surface as collateral, so the assembly-table power ledger kept rendering "Required: 10,000.00 Minecraft Joules" / "Stored: 2,158.09 Minecraft Joules" with the toggle on. New layout: a private `formatMjAmount(double mj, int decimals, boolean abbreviate)` helper centralises the branching (≥ 1000 → `formatAbbreviated` → "10.0k"; otherwise raw `formatDouble`). `localizeMj` and `localizeMjFlow` both route through it with `shouldAbbreviate()` flag; the JEI category labels in `AssemblyTableCategory` and `DistillerCategory` keep calling raw `formatDouble(mj, 1)` directly, so the 80,000-MJ recipe-cost label stays at full precision while the in-game ledger compacts. **What was wrong (fluid side)** — `formatFluidFlow` routed through `formatLong`, which abbreviates with k/M/G/T suffixes on top of the existing unit; a golden-pipe-rate 1600 mB/s rendered as the awkward "1.6k mB/s" instead of the natural-reading "1.6 buckets per second". Real fluid systems read in buckets once values cross that threshold (industrial mods all do this), so the abbreviation branch for fluids now shifts the unit instead of appending a k: ≥ 1000 mB → divide by 1000, unit becomes "buckets" / "B", with k/M/G/T applied on top of the bucket count only if the bucket count itself crosses 1000 (so a 2,000,000 mB/s pipe reads "2.0k B/s"). Toggle off keeps the historical "1,600 mB/s" raw form with the configured thousands separator. **Implementation** — new private `formatFluidAmount(long mb, boolean fullSuffix, boolean abbreviate)` helper does the bucket-shift branching; package-visible `formatMjFlow` and `formatFluidFlow` both gain an explicit `boolean abbreviate` parameter (existing flow tests pick up `, false` to keep their non-abbreviated expectations). Six new tests cover the abbreviation paths against MJ scalars, MJ flow, fluid bucket-shift at the 1000-mB threshold, fluid BOTH-mode where per-sec abbreviates while per-tick stays raw, fluid-toggle-off raw fallback, and fluid k/M/G/T-on-top-of-buckets at very large rates; full `LocaleUtilNumberFormatTester` suite now sits at 34 tests all green.
- Renamed the "Spell Energy Units In Full" config to **"Spell Units in Full"** and broadened it to cover (a) the per-second / per-tick time suffix, matching 1.12.2's spelled-out wording, and (b) fluid pipe tooltips. With the toggle on (default), MJ/RF/FE flow readouts render as "100.00 Minecraft Joules per second" / "5.00 Forge Energy per tick" instead of "100.00 Minecraft Joules/s" / "5.00 Forge Energy/t", and the fluid pipe tooltip reads "40 millibuckets per tick" instead of "40 mB/t". Config-side: `BCEnergyConfig.useFullEnergyNames` → `BCEnergyConfig.useFullUnitNames` (the config-file key changes too, so anyone who had explicitly set `useFullEnergyNames=false` will silently revert to the default `true` on upgrade — acceptable in beta, and the rename matches the broader scope of "all unit display", not just energy). The public `localizeMjFlow` / `localizeRfFlow` / `localizeFluidFlow` wrappers all pass `BCEnergyConfig.useFullUnitNames` so every existing caller (engine ledger, dynamo screen, RF engine screen, energy/fluid pipe tooltips, JEI cost labels) picks up the new behaviour without touching their call sites. The internal `shouldUseFullNames()` helper now backs `mjUnit()`/`energyUnit()` too — same toggle drove unit names already, this just collapses the duplicated null check. Six new unit tests cover the spelled-out form and all three FlowDisplay modes against MJ, RF/FE, and fluid codepaths; the existing seven flow tests pick up `, false` arguments to keep their original "/s"/"/t" expectations. Lang key `buildcraftunofficial.configuration.useFullEnergyNames` → `buildcraftunofficial.configuration.useFullUnitNames` with label updated to "Spell Units in Full".
- Final tube rendering tuning: epsilon-offset the laser start by 1mm below the well's bottom face, restore `partialTicks` interpolation for smooth descent animation, and re-enable per-face diffuse shading. **The earlier integer-snap fix was wrong** — it eliminated the flicker but lost the smooth descent animation entirely, and the diagnosis was incomplete. The user clarified the real issue: the topmost segment was flickering between "world shading" (gradient including the dark interior of the well block) and "block shading" (uniform bright from the open block below the well), with no texture-seam involvement. Tracing through the start-elastic config, the start segment's top vertex sits at `y = well.y` exactly — `BlockPos.containing` resolves that to the **well block itself**, an opaque block with block-light = 0 in its interior. When the start segment is full (~middleWidth tall, length just below an integer), the quad's lightmap interpolates from `light(well-block-interior)` at top → `light(below-well)` at bottom: dark → bright gradient. When `numMiddle` increments and the start collapses to a sliver, the visible first block is occupied by a new middle whose top vertex is just below the well block — both vertices in the bright below-well block, uniform lightmap. Same world pixel, two different lightmap-interpolation states, flips each `numMiddle` increment as `length` crosses an integer. **The fix** — `Vec3 start = new Vec3(pos.getX() + 0.5, pos.getY() - 1e-3, pos.getZ() + 0.5)` in [TubeRenderer](src/main/java/buildcraft/factory/client/render/TubeRenderer.java). The 1mm offset pushes the start segment's top vertex out of the opaque well block and into the bright below-well block. Now both segmentation configurations sample lighting from the same block, the alternation can't happen, and the smooth `partialTicks` animation is back. The 1mm-thin strip at exactly `y = well.y` is no longer rendered, but it was inside the opaque well block's footprint anyway — invisible either way. Also re-enabled `enableDiffuse=true` on the tube's `LaserData_BC8` — this was disabled earlier on the assumption that per-face diffuse plus per-vertex world-light was contributing to the flicker, but they're independent mechanisms: per-face diffuse computes one brightness per quad from the laser's rotation matrix (constant per-laser-render, doesn't depend on `numMiddle`), while the per-vertex lightmap samples world block-light at each vertex's actual position. Restoring diffuse gives the tube vanilla-style block shading (sides at ~0.6/0.8 brightness based on which world-horizontal direction they face, caps at 1.0/0.5 for up/down) without reintroducing any of the segmentation-dependent flicker. End state: smooth length easing, per-face block-like shading, per-vertex world-light gradient down the column, no frame-to-frame artifacts.
- Pruned stale TileMiner entries from [TubeRenderer's](src/main/java/buildcraft/factory/client/render/TubeRenderer.java) `ACTIVE_MINERS` set, fixing per-frame flicker at the top tube segment after world reload. **The symptom** — observed on the mining well: a subtle alternating-frame visual difference in the tube segment directly under the well, looking like two slightly different render submissions competing each frame. The flicker became more obvious after the previous lighting fix because the per-vertex world-light + segmented start-segment math means even small geometry differences are now visible (with the broken fullbright defaults from earlier, every segmentation state looked identical so the duplication was invisible). **Root cause** — same WeakHashMap orphan-leak pattern as the laser renderer (see commit `885f8e027` for the parallel `RenderLaser` fix): client-side level teardown on logout / dimension change doesn't reliably call `setRemoved()` on every BE, so the *old* world's `TileMiner` lingers in the static set until GC runs while the *new* world's `TileMiner` for the same well also registers via `onLoad`. Both render their tube each frame at the same world position but with subtly different `currentLength` smoothing state — the old miner's lastLength/currentLength are frozen at whatever they were the tick before the previous world unloaded, while the new miner's are converging fresh toward the saved `wantedLength`. Two beam submissions per frame at slightly-different lengths means the segmentation boundary (where `numMiddle` increments — see [CompiledLaserType.bakeFor](src/main/java/buildcraft/lib/client/render/laser/CompiledLaserType.java)) is in a different place for each, and the depth test alternates between them frame-to-frame. The flicker concentrates at the top segment because that's the "elastic" segment in our tube's `LaserType` config (`new LaserType(cap, middle, [middle], null, cap)` — `start=middle` makes the start segment absorb the leftover length, while `end=null` leaves the bottom fixed-size), so it's the one whose geometry diverges most between the two miner instances' length values. **The fix** — same one-liner pattern as `silicon.client.RenderLaser`: `ACTIVE_MINERS.removeIf(miner -> miner.isRemoved() || miner.getLevel() != mc.level)` at the top of `onRenderLevel`, plus removing the now-redundant `if (miner.isRemoved()) continue` inside the loop. Self-heals each frame: stale entries get dropped the moment their level reference doesn't match `mc.level`. The previous bullet's `(false, false, 0)` lighting choice stays — that's still correct for the tube's intended look (smooth world-light gradient along the column without per-face brightness jumps); the flicker was a separate lifecycle bug riding on top of it that the lighting change just made visible. **Why this wasn't caught earlier** — pre-port the tube was fullbright everywhere via the broken 4-arg constructor default (`minBlockLight=15`), which meant even with two miners rendering, both produced identical fullbright pixels so the duplication was invisible. The lighting fix in `ecace8685` exposed it by making lighting depend on per-vertex position, and the per-beam override in `885f8e027` doubled down on per-vertex lightmap sampling.
- Per-beam lighting overrides on top of the restored 4-arg defaults: assembly-table lasers back to fullbright, mining well / pump tubes get world-lit gradient without per-face brightness jumps. **Why the previous bullet's fix wasn't quite right** — flipping the 4-arg default to `(true, false, 0)` made the mining well tube *look right per-vertex* (darker as it extends underground) but introduced a separate visual issue: with `enableDiffuse=true`, [LaserContext.setFaceNormal](src/main/java/buildcraft/lib/client/render/laser/LaserContext.java) pre-computes a single diffuse factor per face from a vanilla-style normal map (1.0 top, 0.5 bottom, 0.8 sides), so each of the tube's four sides is uniformly tinted to a different brightness. Stacked against the per-vertex world-light gradient that runs *along* the tube's length, the result is two visual effects in tension — a smooth gradient down the column AND discrete brightness jumps between the visible faces. As the camera rotates around the tube, the dominant visible face flips and the apparent brightness snaps between 0.5x and 1.0x, which the user accurately described as *"as if two rendering systems are competing, one has gradient shading, and the other has more averaged shading"* (they're both real effects from the *same* renderer, not actual duplicate rendering). For the assembly-table laser specifically, the previous bullet's switch to `enableDiffuse=true` also broke the user's expected look — lasers are self-illuminating and the pre-port appearance was fullbright and unshaded, which the world-light + diffuse combination doesn't reproduce. **The fix** — keep the 4-arg default at 1.12.2's `(true, false, 0)` so it still serves the quarry frame / drill / builder build-line / robot beam callers correctly (those *want* the face-shading look), and override at the two callers that need different behaviour: (a) [silicon.client.RenderLaser](src/main/java/buildcraft/silicon/client/RenderLaser.java) explicitly passes `(false, false, 15)` — no face shading, fullbright, matches the pre-port assembly-table laser appearance; (b) [factory.client.render.TubeRenderer](src/main/java/buildcraft/factory/client/render/TubeRenderer.java) explicitly passes `(false, false, 0)` — no face shading, world block-light per vertex. The tube now darkens smoothly along its length as it descends into dark rock (the desired effect) without the per-face brightness jumps that produced the camera-angle-dependent flicker. Result: the mining well tube reads as a single continuous tinted beam; the laser reads as a self-illuminating blue beam at full brightness regardless of ambient.
- Restored 1.12.2 lighting defaults for the 4-arg `LaserData_BC8` convenience constructor — fixes the mining well / pump tube rendering at full-bright the whole way down (instead of darkening with depth) and unshaded (no face-normal diffuse). **The bug** — observed on a mining well tube: as the drill extended through dark underground chunks, the tube stayed uniformly fullbright with no shading variation between faces, looking flat and out-of-place against the surrounding stone. **Root cause** — [LaserData_BC8](src/main/java/buildcraft/lib/client/render/laser/LaserData_BC8.java)'s 4-arg convenience constructor delegated to the 7-arg form with `(false, false, 15)` for the trailing `(enableDiffuse, doubleFace, minBlockLight)` triple. 1.12.2 used `(true, false, 0)` for the same defaults. `minBlockLight = 15` short-circuits [LaserRenderer_BC8.computeLightmap](src/main/java/buildcraft/lib/client/render/laser/LaserRenderer_BC8.java) (line 99 `if (minBlockLight >= 15) blockLight = 15`) to force fullbright on the block-light channel, ignoring world block-light entirely; `enableDiffuse = false` disables face-normal-based shading. - Removed the broken-since-day-one BE renderer for the laser, kept the level-stage one as the single render path, and merged 1.12.2's visual polish into it. **What the dual-renderer investigation actually found** — `TileLaser` had two render paths registered: a `BlockEntityRenderer<TileLaser>` at `silicon.client.render.RenderLaser` (the "proper" architecture, registered via `EntityRenderersEvent.RegisterRenderers`, looking like a faithful port of 1.12.2's `FastTESR<TileLaser>`), and a `RenderLevelStageEvent.AfterTranslucentBlocks` handler at [silicon.client.RenderLaser](src/main/java/buildcraft/silicon/client/RenderLaser.java) (custom, with a static `WeakHashMap`-backed `ACTIVE_LASERS` set populated from `TileLaser.onLoad`). The first-pass conclusion was "delete the level-stage one, the BE renderer is the architecturally correct choice" — but it turns out the BE renderer **has never rendered anything**. Its `submit()` passed *world* coordinates to `LaserData_BC8` along with `Vec3.ZERO` for the camera offset, but vanilla pre-translates the BE pose stack by `(blockPos − cameraPos)` before calling `submit()`, so every vertex ended up at `worldCoord + (blockPos − cameraPos)` — i.e. off by `blockPos` (roughly ±320 blocks for the user's spawn coords, far outside the view frustum). The level-stage renderer was the only thing producing visible beams the entire time; the BE renderer's contribution was silently zero geometry. The "wasted draw calls" claim from the previous bullet was wrong — there was only ever one beam being drawn. **Why the BER path was wrong in the first place** — confirmed by [RenderQuarry's source comment](src/main/java/buildcraft/builders/client/render/RenderQuarry.java) (lines 44-46): *"world-space rendering (frame/drill lasers, rigs) is still handled by BCBuildersEventDist via RenderLevelStageEvent because those cross block boundaries; this BER only covers the block-local LEDs."* The convention across this codebase is BER for block-local geometry, `RenderLevelStageEvent` for anything that spans blocks. The laser beam crosses up to 6 blocks of horizontal range; it belongs in the level-stage path, same as the quarry frame lasers and pump/well tubes. The BE renderer was a port-time misjudgement that no-one caught because the visible-rendering path (the level-stage one) made the absence invisible. **The fix** — delete the broken BE renderer (`silicon.client.render.RenderLaser` + its companion `LaserRenderState` render-state class), remove the `event.registerBlockEntityRenderer(...)` call in [BCSiliconClient.registerRenderers](src/main/java/buildcraft/silicon/client/BCSiliconClient.java) (the whole method drops since that was its only line), and port 1.12.2's `FastTESR<TileLaser>` visual logic *into* the level-stage [RenderLaser](src/main/java/buildcraft/silicon/client/RenderLaser.java): (a) `avg > 200_000` mj/t threshold (`continue` if below — no beam when the laser is essentially idle), (b) direction-aware emitter offset using `BlockLaser.FACING` so the beam emerges from the red face instead of the block centre, and (c) linear `POWERS[(int) (avg * MAX_POWER / maxPerTick)]` colour quantization clamped to `MAX_POWER`, matching 1.12.2's smoother colour transitions instead of the previous coarse 0.25/0.5/0.75 ratio buckets. The orphan-prune `removeIf(laser -> laser.isRemoved() || laser.getLevel() != mc.level)` from the previous bullet attempt stays — that *is* a real bug in the level-stage pattern (client-side level teardown on logout/dimension-change doesn't reliably call `setRemoved()` on every BE; the `WeakHashMap` keeps orphan TileLasers alive past world reload until the next GC sweep, producing visible ghost beams stuck at their pre-save `laserPos`). **Decision rationale** — `TubeRenderer` and the quarry frame/drill renderers both use the same level-stage pattern for the same reason (cross-block beam geometry), and the laser now joins them in being consistent with the codebase convention. `TubeRenderer` still has the same orphan-leak shape (`WeakHashMap` populated in `addMiner`, not pruned per-frame); flagging as a separate follow-up — fixable with the same one-liner `removeIf(miner -> miner.isRemoved() || miner.getLevel() != mc.level)`, but the bug is much harder to observe in practice because pump/well tubes only render while actively extending and the orphan miner's `getLength(partialTicks)` won't keep producing meaningful values once its level reference is dead. **End-to-end check** — full-power laser, save, exit, re-enter: beam comes back blue immediately (per the persistence fix below), single beam, emitter at the red face, debug overlay shows `active_lasers = 1`.
- Laser beam colour now persists across world save/reload, matching 1.12.2. **The bug** — a laser running at full power (blue beam) saved and reopened would come back as a low-power red beam and take ~5 seconds to climb back to blue, even though the recipe at the target table picked up exactly where it left off (so the laser was clearly still pushing the same MJ/t). **Root cause** — [TileLaser](src/main/java/buildcraft/silicon/tile/TileLaser.java) holds a 100-sample `AverageLong` ring buffer (`avgPower`) and a per-tick snapshot field (`averageClient`) which is what [RenderLaser.getLaserTypeForPower](src/main/java/buildcraft/silicon/client/RenderLaser.java) reads to pick `POWER_LOW/MED/HIGH/FULL`. The port saved only the snapshot value (`output.putLong("average_client", averageClient)`), but `serverTick` *unconditionally overwrites* `averageClient = (long) avgPower.getAverage()` on every tick — so the saved value was thrown away the instant the laser ticked, and `avgPower` itself was freshly constructed empty, meaning the average ramped back up from zero over the buffer's 100-tick window regardless of actual power flow. 1.12.2's [TileLaser](https://github.com/legoj15/BuildCraft/blob/8.0.x-1.12.2/common/buildcraft/silicon/tile/TileLaser.java) saved the *whole* `avgPower` ring buffer via `avgPower.writeToNbt(nbt, "average_power")` and restored it via `avgPower.readFromNbt(nbt, "average_power")`; on reload the moving average was already at the right value, so the visual was steady from frame zero. **The fix** — swap the `average_client` long for an `avg_power` `CompoundTag` storing the full buffer (the existing `AverageLong.writeToNbt`/`readFromNbt` helpers on the class are unchanged from 1.12.2; just stop calling them through a tag-codec wrapper since the new `ValueOutput` API takes `CompoundTag.CODEC` the same way the laser's `battery` field already does). On load, derive `averageClient` from `avgPower.getAverage()` so the client-side snapshot reflects the restored buffer before the first server tick fires. The unconditional `avgPower.tick() … averageClient = avgPower.getAverage()` in `serverTick` is left alone — that's the right steady-state behaviour, and now it picks up a populated buffer instead of an empty one. Old worlds saved with the buggy code have no `avg_power` tag and will still see the 5-second ramp-up on their *first* reload after the fix, but subsequent saves preserve state correctly — no save migration needed. The sister field on [TileLaserTableBase](src/main/java/buildcraft/silicon/tile/TileLaserTableBase.java) (`avgPowerClient`, used by [LedgerTablePower](src/main/java/buildcraft/silicon/gui/LedgerTablePower.java) for the GUI flow rate) has the same overwrite pattern, but 1.12.2 didn't persist it either, so leaving it as-is preserves the original behaviour rather than introducing a new divergence. End-to-end check: build a laser+assembly table, wire enough engines to drive the laser to `POWER_FULL` (blue), save, exit, re-enter — beam comes back blue immediately instead of red→yellow→blue over five seconds.

- LED rendering, follow-up — fix depth-ordering between adjacent LEDs. The initial untextured-pipeline switch landed on vanilla's `RenderPipelines.DEBUG_FILLED_BOX`, which inherits `DEBUG_FILLED_SNIPPET`'s `DepthStencilState(LESS_THAN_OR_EQUAL, false)` — depth test on, but **depth write OFF**. The wrapping `RenderType` (still named `buildcraft:led`) and the `VIEW_OFFSET_Z_LAYERING` are unchanged — the layering still helps the LED-against-block-face seam at far view distance, and depth-write handles the LED-vs-LED case the layering alone couldn't. The pipeline reuses vanilla's existing `core/position_color` shader pair (no new shader asset on disk needed; NeoForge's pipeline registrar wires up everything from the snippet inheritance). Side effect: the `TRANSLUCENT` blend mode is now slightly more meaningful — if a future LED ever wants alpha < 0xFF it would still need explicit back-to-front sorting because translucent + depth-write is the classic "transparent stuff hides what's behind it" foot-gun — but our current LEDs are all opaque (alpha=0xFF) so blending degenerates to "replace pixel" and the depth-ordering fix is unambiguously correct.

- **Fluid pipes: the actual cause of the "lag spike per pipe-transit" bug** — every call to `PipeFlowFluids.setFluid(...)` was scheduling a full BlockEntity NBT sync via `pipe.getHolder().scheduleRenderUpdate()`, which on the client (`TilePipeHolder.onDataPacket`) re-invokes `level.sendBlockUpdated()` and **marks the pipe's chunk section dirty, forcing a chunk-mesh rebuild**. That's an entire 16×16×16 section re-meshed every time a pipe transitions empty↔non-empty. In normal flow this happens at the head and tail of the flow path (no big deal), but when a downstream sink is at capacity — e.g. a combustion engine with a full fuel tank — fluid bounces against the full sink and oscillates between adjacent pipe sections. Each oscillation cycle is two empty↔non-empty transitions × every pipe in the path, so the chunk rebuilds compound very quickly (the user's repro: distiller feeding gas to a combustion engine whose tank was full, gas visibly bouncing between the engine and the upstream tank, frame-time spikes locked to fluid movement). The `setFluid` call was scheduling the chunk-rebuild specifically to get the new fluid type to the client, but the lightweight `MessagePipePayload` (`NET_FLUID_AMOUNTS`) registered for fluid flow already covers that: `writePayload` writes the fluid registry id whenever `currentFluid` is non-empty, and the per-section amount loop fires whenever `section.amount` differs from `lastSentAmount` — which is exactly the condition that pairs with `setFluid` (empty → non-empty fires when a section receives fluid in `Section.insert/fill`; non-empty → empty fires when `totalFluid` drops to 0 in `onTick`). The pipe's *block model* doesn't depend on fluid state — pipe connections / paint colour / pluggables drive the model, and those have their own `scheduleRenderUpdate` calls — so the chunk-rebuild was pure waste in the fluid path. Initial state sync on chunk load is unaffected: `writeToNbt` still writes `fluid_id` so the full NBT path picks up the fluid type when a new client starts tracking the chunk. Net effect: bouncing-against-full no longer rebuilds chunk meshes, the spike goes away. The earlier per-frame renderer optimisations from the same investigation remain in place (they're still wins, but they were tackling the *symptom* — high allocation rate during fluid transit — rather than the *cause*).
- Engines: two latent perf bugs in the shared engine renderer that compounded the fluid-pipe issue, especially noticeable when a combustion engine is actively burning fuel (the engine model bakes every frame; under load the per-engine cost was visible at any fuel type, but more pronounced when the engine was at higher power stages where the trunk/piston are animating fast). **Removed the per-engine `bufferSource.endBatch()` full flush** in [RenderEngine_BC8](src/main/java/buildcraft/lib/client/render/tile/RenderEngine_BC8.java) — the previous code called the no-argument `endBatch()` which walks every registered `RenderType` in `MultiBufferSource.BufferSource` and drains its builder. With multiple engines visible (typical pump-room scene: combustion engine + several creative or stirling engines), this forced N full-pool flushes per frame and turned what should be a single batched draw call into a stream of separate uploads + GPU syncs. The `BlockEntityRenderDispatcher` calls `endBatch` on the shared buffer source at the natural end of all BE rendering anyway, so the per-engine call was both redundant and harmful — quads written to the cutout block buffer simply get drained at the correct time. **Made `MutableQuad.setCalculatedDiffuse` allocation-free** in [MutableQuad.java](src/main/java/buildcraft/lib/client/model/MutableQuad.java) — the prior implementation routed through `getCalculatedNormal()` which allocated 7 `Vector3f` per call (four from `MutableVertex.positionvf()` constructors plus three explicit `new Vector3f(...)`), and the engine renderer's loop calls it on every emitted quad. Now the edge vectors and cross product are computed inline as `float` primitives and the result is fed straight into `diffuseLight(x, y, z)`. Visual output identical (same diffuse value, same per-vertex colour set). Every quad-emitting renderer in the project picks up the saving — pipe wires, plug renderers, the builders ghost preview, the laser table, fluid pipes — but engines benefit the most because their bake produces 22 face-quads per engine per frame and each one went through `setCalculatedDiffuse` in the render loop.
- Fluid pipes: kill the per-frame allocation storm in the fluid flow renderer that was producing constant high frametimes (in the order of 40-70 ms per frame at 19 FPS on the test setup — a distiller feeding a combustion engine with fuel + water both in transit) and visible "flickering" between adjacent pipes as the GC pauses fell across renders out of phase with each other. Three layered fixes, all isolated to fluid pipe rendering plus one shared lib helper used by every renderer in the project. **Sprite/tint cache** ([PipeFlowFluids.java](src/main/java/buildcraft/transport/pipe/flow/PipeFlowFluids.java), [PipeFlowRendererFluids.java](src/main/java/buildcraft/transport/client/render/PipeFlowRendererFluids.java)) — the previous renderer rebuilt the still-texture `Identifier`, RGBA tint colour, and translucent-vs-cutout flag every frame for every fluid pipe in view, which under fuel transit (fuel has 3 heat-state Fluid singletons per variant, each with its own sprite path) was three string-built `Identifier.parse`/`fromNamespaceAndPath` allocations + a `FluidStateModelSet` registry lookup + a try-catch'd dispatch per pipe per frame. The eight render-only fields (`renderCacheFluid`, `renderCacheSpriteId`, `renderCacheTint{R,G,B,A}`, `renderCacheTranslucent`) live on the flow as `transient`, populated only when the `Fluid` singleton reference differs from the cached one; the atlas sprite itself is still looked up per frame via `atlas.getSprite(cachedId)` so atlas reloads (resource pack swaps, F3+T) self-heal without an explicit listener. Field types are shared-side classes (`Fluid`, `Identifier`) so the dedicated server build still compiles and never touches the fields. **Scratch-buffer interpolation** — the old `getAmountsForRender(float)` allocated a `double[7]` per call and `getOffsetsForRender(float)` allocated a `Vec3[7]` plus three transient `Vec3` per section (two `Vec3.scale` + one `Vec3.add` = 21 short-lived `Vec3` per call); both are now `writeAmountsForRender(float, double[])` and `writeOffsetsForRender(float, double[], double[], double[])` taking caller-supplied scratch, with `ThreadLocal<double[]>` storage on the renderer reused across every pipe in the render pass. **Direct vertex emit** (the big one) — the inner `renderFluidCuboid` triple loop was the single largest allocator. Per face-cell pair it allocated: one `AABB` (`uvBox`), one `Vector3f` per centre and per radius (×2), and one `MutableQuad` via `ModelUtil.createFace`, which internally allocates the 4-element `Vector3f[]` from `getPointsForFace`, four `Vector3f` from `getPoints`'s `new Vector3f(centerFace)` initialisers, and four more from `addOrNegate(faceRadius, …)` — call it ~13 `Vector3f` plus 1 `MutableQuad` per face per cell. Then `MutableQuad.render` allocated another 2 `Vector3f` per vertex × 4 vertices = 8 more `Vector3f` for the pose-transform scratch. With ~7 cuboids per pipe × ~6 face-cells per cuboid × (~21 Vector3f + 1 MutableQuad + 1 AABB) the per-pipe-per-frame allocations were in the high hundreds, and at 60 fps × ~30 visible pipes the resulting ~1.5 MB/s pure `Vector3f` churn was producing G1 evacuation pauses landing right inside the render thread. The new `renderFluidCuboid` carries geometry as primitive `double`s (no `Vec3`, no `AABB`, no `UvFaceData`), emits vertices directly to the `VertexConsumer` via a per-face dispatch table (no `MutableQuad`), and uses `ThreadLocal<Vector3f>` scratch for the pose-transform pair (no per-vertex allocations). The per-face dispatch table is the hand-traced equivalent of `ModelUtil.createFace` after inlining `getPointsForFace` → `getPoints` → `addOrNegate` for each of the 6 face directions; for each face the table picks the correct cuboid corner positions for vertices 0..3 and pairs each with the right UV corner so the back-face winding stays identical to the old path. The UV mapping switch in `renderFluidCuboid` mirrors `ModelUtil.mapBoxToUvs` directly. **Shared lib helper:** `MutableQuad.renderVertex` ([MutableQuad.java](src/main/java/buildcraft/lib/client/model/MutableQuad.java)) now uses two `ThreadLocal<Vector3f>` scratches instead of allocating per vertex — fluid pipes don't go through that path anymore but every other quad-emitting renderer in the project does (pipe wires, item flow render, plug renderers, builders ghost preview, the laser table, …), so the win is shared. **Regression coverage:** [PipeFlowRendererFluidsGeometryTester](src/test/java/buildcraft/transport/client/render/PipeFlowRendererFluidsGeometryTester.java) drives `emitFace` for all 6 facings on a unit cube against a recording `VertexConsumer`, asserting position and UV for each of the 4 emitted vertices match the hand-traced expected values (themselves derived by walking `ModelUtil.createFace`'s 0..3 vertex assignment under both invert and non-invert paths) — guards against silently regressing the back-face winding or the per-face UV mapping in any future refactor. Plus a separate test verifies the normal vector matches the face step vector for all 6 facings, so lighting and culling stay correct.
- Pump: tighten the "infinite water source" detection so the verdict is per-position under vanilla's own regen rule instead of a once-per-BFS global flag tripped by any qualifying block somewhere in the explored volume. The pre-fix `buildQueue0` walked the flood-fill, and the first block it found with `count >= 2` (orthogonal-or-up source neighbours via `SEARCH_NORMAL`) plus a water-or-solid block below set `isInfiniteWaterSource = true` and broke out of `outer`; the cached flag was then the sole gate on whether `mine()` called `BlockUtil.drainBlock(level, currentPos, true)` to actually remove the source. The pathological case is a horizontal 1×3 strip `[S1][S2][S3]` resting on stone: only the centre `S2` would regenerate after a vanilla bucket scoop (it has S1 and S3 as horizontal source neighbours), but the pre-fix BFS hit `S2` at iteration two with `count = 2` and flipped the global, so a pump positioned over either edge preserved every block of the strip forever rather than consuming all three for three buckets total. New behaviour: a static helper `TilePump.isInfiniteSourceAt(Level, BlockPos)` evaluates the vanilla rule at a single position — `>=2` horizontally-adjacent source-water neighbours via `Direction.Plane.HORIZONTAL` (deliberately not UP/DOWN — a source above does not seed a regenerated source below; water just flows down through air) plus a non-air supporting block below via the existing `getFluidWithFlowing(below) == WATER || below.isSolid()` predicate (water below is kept as valid support to recognise "1×3 strip floating on a deeper pool" as infinite at its centre). `buildQueue0` precomputes `targetPosIsInfinite = isWater && isInfiniteSourceAt(level, targetPos)` once at the top — the anchor block is the tube's landing position, set in `buildQueue` by walking down from the pump until the first non-empty/non-tube block — and only short-circuits the BFS when that one-shot check is true; if it's false (e.g. tube over the edge of a 1×N strip), the BFS now fully explores the connected source volume so the finite tail blocks actually land in `queue` and get consumed instead of being silently skipped past by the early-exit. The inner `count` variable is dropped entirely since the BFS no longer makes its own infinite verdict. The consumption gate in `mine()` re-evaluates the helper against `targetPos` each drain rather than reading the cached flag, so any state change since the queue was built — player scooped a neighbour with a bucket, mob placed a block in the strip, vanilla flow redistributed sources — flips the pump back to consume mode on the very next tick instead of having to wait for a queue rebuild that may never come (a pump's queue rebuilds on drain-attempt failure or config bump, neither of which fires while the pump is happily sitting on a "preserved" infinite source). The class field `isInfiniteWaterSource` is kept but now reflects the latest drain's verdict rather than a sticky once-set BFS flag — the debug-info ledger row `infinite = ...` still works. Knock-on effect that wasn't in the original report but falls out of the same change: a long 1×N strip with the pump over an edge now drains end-to-end correctly. Pre-fix the BFS short-circuited at iteration 2 once it reached an interior block with `count >= 2`, so for `[S1][S2][S3][S4][S5]` with the pump over `S1` only `{S1, S2, S3}` ever made it into `queue`; even with the consumption gate flipped to "finite", `S4` and `S5` stayed orphaned in the world. With the BFS now gated on `targetPosIsInfinite`, an edge anchor falls through to full exploration up to `pumpMaxDistance` (default 64), so the entire connected strip gets queued and consumed. Performance under the original ocean-edge case is unchanged: corners of a 5×5+ pond have ≥2 horizontal source neighbours so `targetPosIsInfinite` is true, BFS still breaks after the first outer iteration (queue ends up with the anchor plus its 5 neighbours from `SEARCH_NORMAL`), and the pump still sits on `currentPos` indefinitely — the only cost is one extra `isInfiniteSourceAt` call per drain (four `getFluid` lookups + one `getBlockState` + one `getFluidWithFlowing` on the below block, all cheap). `pumpsConsumeWater = true` continues to bypass everything (the isWater gate short-circuits to false, so `targetPosIsInfinite` is false and every drain consumes); `CONFIG_REVISION`-driven rebuilds from the recent live-config-toggle fix are unaffected. Eight `PumpInfiniteDetectionTester` game-test cases exercise the helper directly against a `ServerLevel`: 1×3 strip with stone floor verifies S2 infinite + S1/S3 finite; isolated 1×1 source returns false; every corner of a 2×2 pond returns true; 1×5 strip verifies interior S2/S3/S4 infinite + edges S1/S5 finite; a 1×3 strip with no support below returns false at the centre (water without support can't sustain a new source); two diagonally adjacent sources sharing one common corner both return false (regen is orthogonal-only); a 1×3 strip floating on a deeper water layer returns true at the centre (water counts as support); and null-safety short-circuits both arguments so the per-drain hot path won't NPE during tile teardown. End-to-end in-client verification (place a pump over the centre vs. an edge of a hand-dug 1×3 strip, watch the tank fill and the source blocks either remain or vanish) is unchanged and remains the realistic acceptance signal — the helper test is the regression guard against ever silently reverting to the global-flag heuristic.
- Mob pathfinding: pipes, quarry frames, tanks, mining tubes, and laser tables (Assembly/Advanced Crafting/Integration) now report `isPathfindable(LAND/AIR) = false`, so vanilla land mobs treat the cell as a solid obstacle. - Reword `general.pumpsConsumeWater` and its label to match what the option actually does, and make a live config toggle propagate to running pumps without a world reload. The setting's lang label "Pumps Consume Infinite Water" and the modern `.comment("Should pumps slowly drain infinite water sources?")` were both written as if the toggle let pumps gradually deplete an infinite-water pool — neither claim is true. The toggle is a binary kill-switch on `TilePump`'s infinite-source optimisation: with `false` (default), `buildQueue0` flips an `isWater` flag that, on hitting any water cell with ≥2 water-source neighbours and water-or-solid below (the same predicate `BlockDynamicLiquid.updateTick` uses to decide whether to spawn a new source), sets `isInfiniteWaterSource = true` and breaks out of the BFS — the pump then keeps adding 1000 mB to its tank per drain cycle without calling `BlockUtil.drainBlock(level, currentPos, true)`, so the source block stays put; with `true`, the BFS no longer flips that flag and `drainBlock(..., true)` runs on every cycle, actually setting the source block to AIR. There is no "slow" mode — the drain rate is identical either way, gated by the same `10 * MjAPI.MJ` per cycle in `mine()`. There is no progressive depletion either: vanilla MC's flow rules immediately re-source any 2×2+ pool from its remaining source neighbours, so a player toggling the option ON in vanilla and watching a pond expecting it to run dry sees the pool stay full forever, which is what prompted the "doesn't appear to be working" observation. The 1.12.2 comment was clearer about this — *"Should pumps consume water? Enabling this will disable minor optimisations, but work properly with finite water mods."* — and that second clause (the actual reason the toggle exists) got lost during the 26.1 port. Lang label flips to "Pumps Drain Water Source Blocks" and the Java comment is rewritten to explain the optimisation it controls and to call out that the option has no observable effect with vanilla water — it only matters as compatibility scaffolding for mods that disable infinite-water regeneration. Separately, the consume→infinite direction (toggling `true` → `false`) was failing to take effect until the world was saved and reopened, even though NeoForge's `ModConfigSpec.BooleanValue.set(...)` updates the cached value synchronously when the in-game config screen writes the toggle. Cause was state confined to `TilePump` itself: `isWater` is captured per BFS into `buildQueue0`'s local frame, and `isInfiniteWaterSource` is only re-evaluated against the live config in `mine()` at the post-drain `&= !pumpsConsumeWater.get()` line — a pump in consume mode whose `paths` map has been continuously fed by vanilla refill never trips that line in a way that flips back to infinite mode (`&= true` is a no-op on a `false` flag), and a pump whose tank is more than half full short-circuits out of `mine()` at the top guard before reaching either re-evaluation, leaving the cached BFS verdict stuck. World save+reopen "fixed" it incidentally because the BlockEntity is reconstructed with `queueBuilt = false` and `isInfiniteWaterSource = false` (neither field is in NBT), and the first post-reload tick runs `buildQueue` afresh against the current config. Fix: add a static `AtomicLong CONFIG_REVISION` on `TilePump` plus an instance `builtAtRevision` field, bump the static counter from a `ModConfigEvent.Reloading` listener registered in `BCFactory.init` (filtered to `BCUnifiedConfig.SPEC` so a CLIENT-scope display toggle doesn't churn pumps), and gate `serverTick`'s rebuild branch on `!queueBuilt || builtAtRevision != CONFIG_REVISION.get()` so any running pump rebuilds its queue on the very next tick after a config edit, regardless of tank state. The revision is captured inside `buildQueue` itself rather than at call sites, so the existing `mine()` rebuild-on-failure path also stays in sync. Mechanics are otherwise unchanged.
