###### Changes since 26.1.x Beta release 6:
- Drop the dead `display.pipes.powerPipeUseOldMjTexture` config option. The toggle's only consumer was `ActionPowerLimit#getSprite`, where it caused RF kinesis limiter gate-action buttons to render with the MJ-themed icon instead of the RF-themed one — a cosmetic affordance with no further reach despite the name implying it changed pipe-block textures. With the option gone, RF limiter actions permanently use `POWER_LIMIT_RF` (the current `false` default) and MJ limiter actions continue to use `POWER_LIMIT` unconditionally. The `BooleanValue` field, the `BCTransportConfig.buildDisplay` method, the call from `BCUnifiedClientConfig`, and the lang label are all removed; the `display` section in the client config now contains only `useRfNaming`. **One-time migration note:** the `[display.pipes]` block disappears from `buildcraftunofficial-client.toml` on next load — any custom value users had set is silently dropped.
- Wire up the `display.useRfNaming` toggle and flip its default to FE. The option had been wholly cosmetic but landed in 26.1.1 with two compounding bugs: it defaulted to `true` (RF) but every user-facing string had already been rewritten in FE/Forge Energy wording, and the only `useRfNaming.get()` call site was a `BlockEngineFE` ternary whose two branches returned the same translation key — toggling did nothing even there. Default flips to `false` (FE matches modern NeoForge convention), and the toggle now actually swaps the displayed nomenclature: the FE Engine block name (custom `BlockItem` overrides `getName(ItemStack)` to pick `block.buildcraftunofficial.engine_rf` vs `.rf` sibling), the nine FE pipe item names (`ItemPipeHolder.getName` detects FE pipes via `definition.identifier.endsWith("_rf")` and adds the same suffix), the engine GUI title (`BlockEngineFE.openGui` now feeds `BCEnergyConfig.rfFeKey(...)` to the `MenuProvider`), the engine/dynamo battery hover (`%,d / %,d FE/RF`), the upgrade-rate per-second tooltip ("Forge Energy"/"Redstone Flux"), the help-overlay "FE Energy"/"RF Energy" titles and conversion bodies, the gate "Switch to N FE/t" / "RF/t" action label, the JEI "FE Transport"/"RF Transport" guide chapter, the engine and MJ-dynamo `tip.block.*` tooltips ("Forge Energy"/"Redstone Flux" wording), and the `LocaleUtil.localizeRf`/`localizeRfFlow` "FE/s"/"RF/s" unit suffixes — all driven by a single `BCEnergyConfig.rfFeKey(baseKey)` helper and a `LocaleUtil.energyUnit()` helper, with the existing translation key kept as the FE branch and a `.rf` sibling added for each affected entry. The `BCTooltips` registry-driven tooltip handler now also runs every key through `rfFeKey` and falls back to the base key when no `.rf` variant exists in the lang file, so non-energy tooltips (`tip.block.engine_stone`, `engine_iron`, etc.) are unaffected. Internal block/item registry IDs (`engine_rf`, `pipe_*_rf`) are unchanged — renaming them would break existing worlds, and the `_rf` suffix is a harmless legacy artefact.
- Config audit follow-up: drop `display.hidePower` and `display.hideFluid`. The previous config-audit pass kept these dead options on the books with `// TODO: wire into tile GUIs` markers, on the assumption they'd be re-wired the way the old pre-license-migration 1.12.2 code did it (gating the textual MJ/mB readouts in every engine/quarry/refinery/pipe-tooltip GUI). Decided not to revive that — the bars and gauges already convey the information, and re-implementing the toggle would mean threading two booleans through every BC tile GUI's `drawString` call for a feature few players asked for. The two `BooleanValue` fields, `BCCoreConfig.buildDisplay`, the call from `BCUnifiedClientConfig`, and the two lang labels are all removed; the `display` section in the client config now contains only `useRfNaming` and `pipes.powerPipeUseOldMjTexture` — both of which actually have consumers.
- Config audit, phase 1 (cleanup, no behaviour change). Removes three orphan section keys (`events`, `experimental`, `performance`) that survived the 1.12.2→26.1.1 unification but no longer correspond to any section in `BCUnifiedConfig` — the in-game config screen had been silently looking them up forever. Adds the three lang labels that were missing (`useRfNaming`, `baseRfRate`, `baseFlowRate`) so those options no longer fall back to camelCase identifiers in the UI. Adds the eleven `.comment(...)` strings that were missing on `BCEnergyConfig` options (oil burn, the four spout heights, the three tier probabilities, both excluded-list-mode booleans, and `oilIsSticky`) so the hover-tooltips on those rows now read as English sentences instead of being blank. Renames the worldgen subsection `worldgen.oil.spawn_probability` → `worldgen.oil.spawnProbability` to match the camelCase used by every other section, and renames the seven SCREAMING_SNAKE Java fields in `BCBuildersConfig` (`QUARRY_FRAME_MIN_HEIGHT`, …, `BPT_STORE_EXTERNAL_THRESHOLD`) to `quarryFrameMinHeight` / `bptStoreExternalThreshold` / etc. so the field naming matches the TOML keys (which were already camelCase) and the rest of the config classes; the six call-sites in `TileQuarry` are updated. The `excessiveBiomes` comment is also rewritten to actually explain what the option does — every biome in the list receives a flat 30× multiplier on small/medium/large oil rolls; in 1.12.2 this defaulted to BC's own oil_desert/oil_ocean biomes (which no longer exist), so the empty default is correct, but the option remains as a "force a vanilla/modded biome to gush oil" override.
- Config audit, phase 2 (translation key paths). NeoForge's `ConfigurationScreen` builds option labels as `<modid>.configuration.<leaf>` — the leaf option name only, with no enclosing-section prefix (verified against `ConfigurationSectionScreen.getTranslationKey` in `neoforge-26.1.2.30-beta-sources`, line 545: `context.modId + ".configuration." + key`). Every `pipes.*` / `oil.*` / `oil.spouts.*` / `oil.spawnProbability.*` prefixed key in `en_us.json` was therefore an orphan, and the corresponding options had been silently falling back to their raw camelCase identifiers in the UI for the entire 26.1.1 lifetime. Strips the prefixes from all 17 affected leaf-option keys (`pipes.mjPerItem` → `mjPerItem`, `oil.spouts.smallSpoutMinHeight` → `smallSpoutMinHeight`, etc.), and adds the two missing sub-section header labels (`spouts` → "Oil Spouts", `spawnProbability` → "Spawn Probability") since NeoForge looks up sub-section buttons by their own leaf name too — previously the `[worldgen.oil.spouts]` and `[worldgen.oil.spawnProbability]` sub-screens opened with camelCase headers. Three oil-tier probability labels also rephrased from "Small/Medium/Large Oil Gen Prob" to "Small/Medium/Large Oil Spawn Probability" while we're rewriting that line anyway. No code change — purely a lang-file fix.
- Config audit, phase 3 (re-categorise mis-placed paths). Five settings sat in sections that didn't match what they actually controlled: `oilIsSticky` was at top-level `general` but is a worldgen.oil concern (every other oil setting lives there); `useRfNaming` was at top-level `general` but is a pure-cosmetic display toggle that only renames the FE engine and pipes; `disableRfPipe` was at top-level `general` but every other pipe option was already nested under `general.pipes`; `bptStoreExternalThreshold` and the six `quarry*` options were at top-level `general` while the rest of the schema was already split by module/feature. Reorganised: `oilIsSticky` moves to `worldgen.oil.oilIsSticky`, `useRfNaming` moves to `display.useRfNaming` (via a new `BCEnergyConfig.buildDisplay` slotted into the unified spec), `disableRfPipe` moves under the existing `general.pipes` push, and `bptStoreExternalThreshold` plus all six quarry options nest under a new `general.builders` (and `general.builders.quarry` for the six quarry knobs). Two new section labels added (`builders` → "Builders", `quarry` → "Quarry"). **One-time migration note:** the keys' on-disk paths in `buildcraftunofficial-common.toml` change with this refactor, so the next world load will regenerate the moved entries at their new paths with default values; users who had customised any of these specific knobs (most commonly `quarryMaxBlockMineRate` or `pumpsConsumeWater`) will need to re-apply those settings under the new sections. All other config values are unaffected.
- Config audit, phase 4 (split CLIENT scope from COMMON). The unified config was registered as `ModConfig.Type.COMMON` for everything including the four `display.*` toggles (`hidePower`, `hideFluid`, `useRfNaming`, `pipes.powerPipeUseOldMjTexture`), which meant the server's choice would be applied to every connected client — wrong scope for per-player UI preferences. Split into two specs: COMMON (gameplay/balance: `general.*`, `worldgen.*`) stays in `BCUnifiedConfig` and registers from `BCCore.init()`; a new `BCUnifiedClientConfig` holds the `display.*` section and registers from `BCCoreClient.init()` as `ModConfig.Type.CLIENT`. NeoForge writes the latter to `config/buildcraftunofficial-client.toml` and serves it through the same Mods → Config screen as a separate "Client" tab. The mods screen now shows two config buttons stacked under a "Common" / "Client" header instead of one. **One-time migration note:** any non-default `display.*` value previously sitting in `buildcraftunofficial-common.toml` will be ignored on next load; users will need to re-apply those settings in the new client toml. The old `[display]` block in the common file is harmless to leave behind but can be deleted manually.
- Config audit, phase 6 (resolve dead settings). Nine config options were defined-but-never-read after the 1.12.2→26.1.1 unification — each had a label, a default, sometimes a comment, but zero `.get()` callers. Three are removed outright because the underlying feature isn't in 26.1.1 at all: `worldGenWaterSpring` (no springs in 26.1.1), `quarryFrameMoveBoth` (the both-axis frame-move animation mode wasn't ported), and `enableOilBurn` (oil flammability isn't implemented in `BCEnergyFluids`). The other six (`worldGen`, `minePlayerProtected`, `miningMultiplier`, `bptStoreExternalThreshold`, `hidePower`, `hideFluid`) gate features that DO ship in 26.1.1 but the runtime path doesn't currently consult the config — they stay defined and labelled, but each gets a `// TODO: wire into <consumer>` comment in its declaring class so the gap is visible at the source rather than only in the audit doc. Nothing user-visible changes from leaving them in this session, but the schema is now honest about which knobs do nothing yet. **One-time migration note:** the three removed keys disappear from `buildcraftunofficial-common.toml` on next load; existing customisations on those keys (typically all defaults) are lost without effect.
- Config audit, phase 7 (naming and range cleanup). Three fixes that don't affect what works but make the schema readable. (1) `excludedBiomesIsBlackList` and `excludedDimensionsIsBlackList` are double-negative booleans where flipping them silently reverses list polarity — replaced with two `EnumValue<ListMode>` (`biomeListMode`, `dimensionListMode`) that surface as a `BLACKLIST | WHITELIST` dropdown in the config UI. The two callers in `OilGenerator` were updated; same default behaviour as before (`BLACKLIST`). (2) `excessiveBiomes` is now `forceExcessiveOilBiomes`, with a comment that explains what it actually does — every biome in the list receives a flat 30× multiplier on small/medium/large oil rolls. The 1.12.2 default of `[buildcraftenergy:oil_desert, buildcraftenergy:oil_ocean]` was the original target (BC's own oil biomes), but those biomes don't exist in 26.1.1 so the modern default is `[]`. The option remains as a "force a vanilla/modded biome to gush oil" override; the rename makes that intent explicit. (3) Oil spout height bounds were `0..256`, which capped below the modern `-64..320` build height; bumped to `-64..320`. **One-time migration note:** users with custom values for `excludedBiomesIsBlackList`, `excludedDimensionsIsBlackList`, or `excessiveBiomes` in their existing TOML will see those entries ignored on next load (they're not in the new spec) and the new keys will appear at their defaults; re-apply customisations under the new names if needed.
- Assembly Table: surface every recipe in JEI under a new `buildcraftunofficial:assembly_table` category, with facades collapsed into a single cycling entry instead of one entry per source block (which would otherwise mint thousands of near-identical JEI rows). Each non-facade recipe in `AssemblyRecipeRegistry.REGISTRY` contributes one entry per `getOutputPreviews()` stack — chipsets, gates, lenses, plugs, the gate copier — with input slots gathered from `AssemblyRecipe.getInputsFor(output)` (`Ingredient.getItems()` × `IngredientStack.count`) and the per-recipe MJ cost from `getRequiredMicroJoulesFor(output)`. The facade path is special-cased: `AssemblyRecipeCollector` walks `FacadeStateManager.validFacadeStates` once and emits a single `AssemblyRecipeJei` whose two input slots and one output slot each carry a list — slot 0 is `3× pipe_structure` (fixed), slot 1 cycles through every visible source block, and the output cycles through `createFacadeStack(info, false)` and `createFacadeStack(info, true)` (basic + hollow); each source block is doubled in the input list to stay paired with its two output variants the way `FacadeAssemblyRecipes.getRecipeInputs/Outputs` already does in-game. JEI's native `addItemStacks(List)` cycling renders this as one browsable recipe — pressing U on Stone surfaces the facade entry (Stone is in the cycling input list), pressing R on a Stone Facade surfaces the same entry from the output side, and the JEI index gains exactly one row for the entire facade family rather than ~5,000. Layout mirrors `GuiAssemblyTable`: an 89×86 crop of `assembly_table.png` at `(3, 27)` provides the input grid and power bar visuals (input slots laid out at the cropped `(5,9)` origin, up to a 3×4 grid filled in row-major order based on how many inputs the recipe actually has — most use 1–2), with a single output slot to the right at `(105, 9)` drawn with JEI's `setOutputSlotBackground()`. The MJ cost renders as `"X.X MJ"` below via the new `gui.jei.category.buildcraftunofficial.assembly_table.power` lang key (matching the distiller's pattern; assembly costs span 500 MJ for a lens to 80,000 MJ for a Gold AND Gate, so the number is load-bearing). Catalyst is `BCSiliconItems.ASSEMBLY_TABLE`. The collector's facade path defensively returns null if `pipe_structure` isn't registered (rather than falling back to the cobblestone-wall placeholder the in-game recipe uses for an empty registry — a JEI entry with the wrong recipe shape would mislead more than it helps), and skips both empty-output recipes and ingredients with no matching items so a misconfigured recipe never crashes the plugin init.
- JEI fluid recipes: extend the invisible-bucket alias system on the distiller and heat-exchanger categories to also surface fragile fluid shards (and any other modded "bucket-like" fluid container that registers itself), so a survival player holding a Fragile Water Shard can U-look-up the distiller the same way they could with a water bucket. The new `buildcraft.lib.compat.jei.FluidContainerAliases` utility owns the provider list, ships built-in providers for the vanilla/NeoForge bucket (`Fluid#getBucket()`) and for `ItemFragileFluidContainer` (constructs an ItemStack with the fluid pre-loaded into the `FLUID_CONTENT` data component), and exposes `registerProvider(Provider)` so other mods can add their own forms (e.g. a hypothetical clay bucket) from their JEI plugin without us needing to scan the item registry — capability scans were considered and rejected because universal tanks would match every fluid recipe and pollute lookups. `DistillerCategory` and `HeatExchangerCategory` now both call into this utility, replacing the verbatim-duplicated `addBucketAlias` private helpers the original distiller-JEI changelog entry below explicitly punted on extracting. To keep the shard alias robust regardless of how full the player's shard is, `BCCoreJeiPlugin.registerItemSubtypes` registers an `ISubtypeInterpreter` for `FRAGILE_FLUID_CONTAINER` keyed on the fluid's resource location alone — `registerFromDataComponentTypes(FLUID_CONTENT)` would have keyed on the whole `SimpleFluidContent` (including amount), so a 250 mB and 500 mB water shard would have hashed to different ingredients and the alias (a 500 mB shard) would only have matched exactly-full shards. Per-fluid shards are intentionally not enumerated into JEI's master ingredient list — shards remain alias-only.
- Distiller: restore 1.12.2's bucket right-click behaviour and the underlying tank gating that makes it work. Three regressions to address: (1) the modern port lost `tankIn.setFilter(isDistillableFluid)`, so right-clicking the distiller with a lava bucket happily fed the lava into the input tank where it would just sit forever; (2) the modern port lost `tankIn.setCanDrain(false)`, so right-clicking with an empty bucket drained the input tank instead of falling through to the outputs — the user's distillable input would silently disappear into the bucket on a misclick; (3) the modern port lost `tankGasOut/tankLiquidOut.setCanFill(false)`, so any pipe or bucket interaction could shove arbitrary fluids into either output tank. Fix mirrors the heat exchanger's `OutputTank` pattern: `TileDistiller_BC8` gains an `InputTank` whose `isValid` defers to `isDistillableFluid` and whose `extract` returns 0 unless an internal-flag-flipped `extractInternal` call is in progress, plus an `OutputTank` (its own subclass; the heat exchanger's lives in a different tile and uses a different capacity) whose `isValid` is gated by an internal-insert flag — the craft loop now drains the input via `extractInternal` and fills the outputs via `insertInternal`, while NBT load and the heat-exchanger-style `loadTank` helper keep using `set()` directly to bypass both gates (saved fluid is trusted, and the recipe registry may not be ready during early chunk load). `BlockDistiller.useItemOn` now iterates `tankIn` → `tankGasOut` → `tankLiquidOut` calling `FluidUtilBC.onTankActivated` on each — the gates funnel each held item to the right tank: a distillable bucket fills `tankIn` (filter passes; outputs reject the insert), an empty bucket drains `tankGasOut` first then `tankLiquidOut` (input rejects the extract), a lava bucket lands nowhere and the GUI opens. Three new regression tests cover the input-tank distillable filter, the input-tank external-extract block (and that `extractInternal` still drains properly so distillation doesn't halt), and the output-tank external-insert block (with `insertInternal` still landing recipe results, and the internal flag resetting after the call).
- Distiller: restore 1.12.2's wrench-rotates-the-block behaviour. Right-clicking the distiller with a wrench was opening the GUI instead of rotating it because `BlockDistiller` neither implemented `ICustomRotationHandler` (so `ItemWrench_Neptune.useOn` → `CustomRotationHelper.attemptRotateBlock` returned `PASS`) nor short-circuited the wrench in `useItemOn`, so the block fell through to its bucket-then-GUI path and the wrench's `useOn` never ran. Mirroring the engine pattern, `BlockDistiller` now implements `ICustomRotationHandler.attemptRotation` to rotate the horizontal `FACING` clockwise (1.12.2's `IBlockWithFacing.attemptRotation` → `EnumFacing.rotateY()`, equivalent to `Direction.getClockWise()` in 1.21+) and returns `PASS` from `useItemOn` whenever the held stack is an `IToolWrench` so the wrench's `useOn` actually fires the rotation, the slide sound, and the `buildcraftunofficial:wrenched` advancement. Empty-hand right-click still opens the GUI; bucket / fluid-container right-click still drains/fills the input tank.
- Distiller: client and server tank states now stay in sync after a server-side drain, fixing the visual desync where a wooden(-diamond) fluid pipe pulling gaseous fuel out of the distiller looked like it was barely extracting a trickle while the gas tank stayed visually full — the pipe was actually draining at the expected rate (~3 mB/t, matching the 16 mB / 5.33 ticks production), the client just never saw it. `saveAdditional` only writes the `fluidIn` / `fluidGasOut` / `fluidLiquidOut` keys when the corresponding tank is non-empty, but `loadAdditional` was using `tank.insert(0, fluid, amount, tx)` which *adds* to the existing client-side stack instead of replacing it: every `getUpdateTag` round-trip topped up the client's stale value (1000 mB → server drains to 800 → client sees 1800 next packet), and an empty server tank (key omitted) left the stale client value in place forever. A world reload "fixed" it because the BlockEntity was constructed fresh from an empty starting state. Fix mirrors the heat exchanger's own `loadTank` helper from the recent post-GUI fixes: a small private helper that always calls `set(0, ...)` — passing `(EMPTY, 0)` when the saved value was empty — so client tanks track every drain. The same `insert`-instead-of-set pattern would have affected anything else that piped fluid out of the distiller (input bucket containers, JEI lookups against the live tank, comparator readouts), but only the gas-output path hit it visibly because that's the only one being drained continuously by an external machine.
- Tank: a creative Ctrl+middle-click on a full tank used to copy the contents onto the picked item, so placing the result spawned a fresh full tank — every placement minted another 16 000 mB of whatever was in the source. Modern Minecraft's pickblock-with-data path runs server-side `addBlockDataToItem`, which calls `BlockEntity#saveCustomOnly` (writes our `tank.serialize`'s `"stacks"` key) → `removeComponentsFromTag` (default no-op) → `BlockItem.setBlockEntityData`, so the tank's full fluid landed in the item's `BLOCK_ENTITY_DATA` component and `BlockItem#updateCustomBlockEntityTag` re-applied it to the new BE on placement. `TileTank` now overrides `removeComponentsFromTag(ValueOutput)` to `discard("stacks")`, leaving the tag empty for the picked-item path so `setBlockEntityData` removes the component entirely. Disk save / chunk reload still go through `saveAdditional`/`loadAdditional` directly and are unaffected — only the pickblock-into-itemstack path drops the fluid payload. BC's intended economy is that tanks must be broken (which already drops fluid as separate fluid-shard items) before being moved, so picked tanks now always come back empty.
- Fluid pipes: restore the 1.12.2 tiered transfer rates that the port was missing, so higher-tier pipes actually move fluid faster than wood. `BCTransportConfig` only registered `PowerTransferInfo`/`RedstoneFluxTransferInfo` for kinesis/RF pipes; the matching `fluidTransfer(...)` calls were dropped, so every fluid pipe fell back to `PipeApi.fluidInfoDefault`'s `(20 mB/t, 10)`. New `baseFlowRate` config (default 10 mB/t, range 1–40) plus a `registerFluidTransferData()` invoked from `FMLCommonSetupEvent` next to the existing power/RF registrations: cobble/wood = base, stone/sandstone = 2× (20 mB/t), clay/iron/quartz = 4× (40), diamond/diaWood/void = 8× (80), gold = 8× with delay 2 instead of 10 (low-viscosity routing). Symptom: a wooden/diamond/gold/etc. fluid pipe between a distiller and a tank stack drained at the same 20 mB/t as a cobble pipe, taking ~40 s per 16 000 mB tank when it should have been a few seconds for the higher tiers — and balance-shifting via tank stacking made the slowdown look like duplication because the new top tank visibly filled "instantly" while the bottom emptied.
- Distiller: light up the GUI's spare overlay assets so the screen reflects machine state instead of staying static. The four spare regions in `distiller.png` were never wired up post-port, so the input/output tanks rendered as solid fluid rectangles with no gauge marks and the central pipe schematic between them showed only the dim background art whether the distiller was running, blocked on a full output, or idle. Three changes: (1) `GuiDistiller` now passes the `(0,161) 16x38` and `(17,161) 34x17` regions as gauge overlays to the input tank and the two output tanks respectively (mirroring what the heat exchanger does with the matching sub-images in its texture), so each tank renders the recipe-coloured fluid behind the proper gauge marks. (2) The bright `(212,0) 36x57` running overlay now blits over the central `(61,12)` schematic region whenever the distiller's `isActive` is true (a recipe is matched and power was actually extracted this tick), and the dim `(176,0) 36x57` "stuck" overlay blits there instead when a new `isStuck` flag is true. (3) `TileDistiller_BC8` tracks `isStuck` as `currentRecipe != null && (!canFillLiquid || !canFillGas)`, set inside the same `serverTick()` block that already evaluates the can-fill checks, cleared back to false when no recipe matches the input, included in the per-tick `needsSync` change check that triggers `level.sendBlockUpdated`, and persisted in `saveAdditional`/`loadAdditional` so a chunk reload preserves the visual state. The two centre overlays are mutually exclusive by construction — `isActive` requires `canFillLiquid && canFillGas`, the same fills `isStuck` negates — so the GUI just picks whichever is true.
- Distiller: surface every registered distillation recipe in JEI as its own entry under a new `buildcraftunofficial:distiller` category, mirroring the in-world `GuiDistiller` layout. Each `IDistillationRecipe` from `BuildcraftRecipeRegistry.refineryRecipes.getDistillationRegistry()` maps 1:1 to a JEI entry — unlike the heat exchanger no pairing is needed because the recipe interface already carries the full operation (one input fluid, one gas output, one liquid output, one MJ cost). The category background is the inner 170×74 region of `distiller.png` starting at `(3, 4)`, masking out the GUI's outer border and the player-inventory section so the JEI panel shows only the machine itself; tank slots are placed at the `GuiDistiller` coordinates shifted by the same `(-3, -4)` offset so they line up with the tank graphics inside the cropped texture (input `(41,19) 16x38`, gas output `(95,6) 34x17`, liquid output `(95,50) 34x17`). The per-recipe MJ cost (from `IDistillationRecipe.powerRequired()`, divided by `MjAPI.MJ` through `double` so sub-MJ recipes don't render as `0`) is drawn as `"X.X MJ"` in the strip below the tanks via the new `gui.jei.category.buildcraftunofficial.distiller.power` lang key, matching the in-GUI label colour `0xFF404040` — without it, two recipes with identical fluids but different power costs would be indistinguishable, and runtime power requirements wouldn't be discoverable from JEI at all. All three slots are real INPUT/OUTPUT (not RENDER_ONLY), so JEI lookups behave as players expect: pressing U on a fluid surfaces every distillation that consumes it; pressing R on a gas or liquid output surfaces every distillation that produces it. Each fluid slot also registers the matching bucket item as an invisible lookup ingredient (same `addBucketAlias` pattern the heat exchanger uses, deliberately duplicated as a private static rather than extracted — two callers don't yet justify a shared utility), so survival-mode lookups on bucket items resolve to the same recipes without adding the bucket to the visible cycle. The enumeration helper `enumerateDistillationRecipes` defensively skips entries with an empty input or with both outputs empty, guarding against config-disabled fluids that the runtime would refuse to run anyway. The distiller block (`BCFactoryItems.DISTILLER`) is registered as the single catalyst; pressing R on it surfaces the new category.
- Heat Exchanger: surface every valid pairing in JEI, drawn as the actual machine in miniature. A single category `buildcraftunofficial:heat_exchanger` enumerates the cartesian product of the heatable and coolable registries at plugin-registration time, keeping only pairs where `coolable.heatFrom() > heatable.heatFrom()` — exactly the runtime pairing rule in `TileHeatExchange.craft()` — and wraps each as a `HeatExchangerRecipePair` JEI entry. The category background is the inner 170×84 region of `heat_exchanger.png` starting at `(3, 4)`, masking out the GUI's outer border and the player-inventory section so the JEI panel shows only the machine itself; all four fluid tanks are positioned at the `GuiHeatExchange` coordinates shifted by the same `(-3, -4)` offset so they line up with the tank graphics inside the cropped texture (`(41,8)` END input, `(41,60)` START input, `(95,8)` END output, `(113,39)` START output). The four fluids are routed across the layout the same way `TileHeatExchange.craft()` routes them at runtime: heatable input at MID-LEFT, coolable input at TOP-LEFT, heatable output at TOP-RIGHT (the runtime `h_out = end.tankOutput`), coolable output at MID-RIGHT (the runtime `c_out = start.tankOutput`). The two flows therefore cross diagonally — the same fluid type sits on each diagonal — exactly as players see it in-world when cooling hot oil with cool oil or water (cool oil enters MID-LEFT, exits heated at TOP-RIGHT; hot oil enters TOP-LEFT, exits cooled at MID-RIGHT). Earlier per-side categories with cycling RENDER_ONLY opposite tanks broke this because the cycle could land on a non-symmetric coolant; an interim paired-recipe attempt put both outputs on the same row, so a water + hot-oil cooling pair drew the cooled cool oil at TOP-RIGHT instead of MID-RIGHT. All four slots are real INPUT/OUTPUT (not RENDER_ONLY), so JEI lookups behave as players expect: pressing U on lava surfaces every entry where lava is the heat source paired with various heatables; pressing U on water surfaces the single entry that pairs water with lava; pressing R on a hot oil fluid surfaces every entry that produces it on either side. Each fluid slot also registers the matching bucket item as an invisible lookup ingredient (`addInvisibleIngredients(role).add(fluid.getBucket())` when the fluid registers a bucket), so pressing R or U on a bucket of cool oil surfaces the same recipes as pressing R or U on the cool-oil fluid entry — what a survival-mode player actually has in hand. The recipe view stays clean — the bucket is invisible, so the slot continues to render as a fluid tank rather than cycling between fluid and item rendering. Recipes whose `out()` is null (water heating, lava cooling) skip the corresponding output slot — the GUI tank background remains visible behind the missing fluid.
