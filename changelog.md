###### Changes since 26.1.x Beta release 6:
- Heat Exchanger: surface every valid pairing in JEI, drawn as the actual machine in miniature. A single category `buildcraftunofficial:heat_exchanger` enumerates the cartesian product of the heatable and coolable registries at plugin-registration time, keeping only pairs where `coolable.heatFrom() > heatable.heatFrom()` ‚Äî exactly the runtime pairing rule in `TileHeatExchange.craft()` ‚Äî and wraps each as a `HeatExchangerRecipePair` JEI entry. The category background is the top 176√ó86 region of `heat_exchanger.png` (cropping just before the player inventory) so the JEI view shows the same pipe schematic, tank shapes, and bucket-slot recesses as the in-world GUI; all four fluid tanks are positioned at the same coordinates `GuiHeatExchange` uses (`(44,12)` END input, `(44,64)` START input, `(98,12)` END output, `(116,43)` START output). The four fluids are routed across the layout the same way `TileHeatExchange.craft()` routes them at runtime: heatable input at MID-LEFT, coolable input at TOP-LEFT, heatable output at TOP-RIGHT (the runtime `h_out = end.tankOutput`), coolable output at MID-RIGHT (the runtime `c_out = start.tankOutput`). The two flows therefore cross diagonally ‚Äî the same fluid type sits on each diagonal ‚Äî exactly as players see it in-world when cooling hot oil with cool oil or water (cool oil enters MID-LEFT, exits heated at TOP-RIGHT; hot oil enters TOP-LEFT, exits cooled at MID-RIGHT). Earlier per-side categories with cycling RENDER_ONLY opposite tanks broke this because the cycle could land on a non-symmetric coolant; an interim paired-recipe attempt put both outputs on the same row, so a water + hot-oil cooling pair drew the cooled cool oil at TOP-RIGHT instead of MID-RIGHT. All four slots are real INPUT/OUTPUT (not RENDER_ONLY), so JEI lookups behave as players expect: pressing U on lava surfaces every entry where lava is the heat source paired with various heatables; pressing U on water surfaces the single entry that pairs water with lava; pressing R on a hot oil fluid surfaces every entry that produces it on either side. Each fluid slot also registers the matching bucket item as an invisible lookup ingredient (`addInvisibleIngredients(role).add(fluid.getBucket())` when the fluid registers a bucket), so pressing R or U on a bucket of cool oil surfaces the same recipes as pressing R or U on the cool-oil fluid entry ‚Äî what a survival-mode player actually has in hand. The recipe view stays clean ‚Äî the bucket is invisible, so the slot continues to render as a fluid tank rather than cycling between fluid and item rendering. Recipes whose `out()` is null (water heating, lava cooling) skip the corresponding output slot ‚Äî the GUI tank background remains visible behind the missing fluid. The heat exchanger block (`BCFactoryItems.HEAT_EXCHANGE`) is registered as the single catalyst. Distillation recipes (`IDistillationRecipe`) are unaffected and remain a follow-up ‚Äî different machine block, different recipe shape (1 in / 2 out + power), so they get their own category in a later change.
- Filtered Buffer: render the empty-filter funnel icon ourselves in `GuiFilteredBuffer.drawBackgroundTexture` instead of via `Slot.setBackground(...)`. The container was passing `BCTransportSprites.EMPTY_FILTERED_BUFFER_SLOT.getResourceLocation()` (`buildcraftunofficial:gui/empty_filtered_buffer_slot`) to `setBackground`, but vanilla's `AbstractContainerScreen.renderSlot` resolves slot backgrounds against the GUI sprite atlas (`Sheets.GUI_SHEET`) only ‚Äî and that sprite lives on the blocks atlas via the `gui/` directory source in `assets/minecraft/atlases/blocks.json`. The atlas miss made every empty filter slot in the top row paint as the magenta/black missing-texture sprite. Drawing it via `GuiIcon.drawAt(EMPTY_FILTERED_BUFFER_SLOT, ...)` routes through `SpriteHolderRegistry`'s multi-atlas walker and finds the sprite on the blocks atlas, matching how the bottom row's `NOTHING_FILTERED_BUFFER_SLOT` already rendered correctly. The `setBackground` call is dropped from `ContainerFilteredBuffer` since `Slot#getNoItemIcon` is the only consumer and the GUI now does the rendering.
- Heat Exchanger: fix four bugs introduced with the recent restore-restrictions / add-GUI work. (1) Each cycle's four tank operations (output fill √ó2, input drain √ó2) now run inside a single transaction that only commits when every operation transferred exactly the expected `min_common` amount; previously each operation opened its own root transaction and committed unconditionally, so any simulation/execution mismatch leaked input fluid (e.g. the user-reported 5mb of lava + 5mb of cool oil left over after processing 2 buckets each into 2 buckets of hot oil). The Distiller already ran its `extract` + two `insert`s under one transaction; this brings the exchanger to parity. Two new `HeatExchangerTester` cases cover the balanced-commit and undersized-fill-rolls-back paths. (2) `WidgetFluidTank` now falls back to filling an empty bucket from the player's inventory when the cursor itself is holding a full bucket of the tank's fluid ‚Äî without this, a single-bucket cursor goes empty ‚Üí full on the first click and then every follow-up click on an output tank does nothing (the tank rejects external inserts via `OutputTank.isValid` and the cursor bucket has no room to grow), making it look like the tank stopped giving up fluid after the first 1000mb. The fallback walks `player.getInventory()`, picks the first slot that exposes `Capabilities.Fluid.ITEM` with an empty resource, and `move`s a bucket-worth from the tank into that slot via a `forPlayerSlot(...).oneByOne()` access; the cursor stays untouched, mirroring the way 1.12.2's in-world right-click happily swallowed buckets in succession. (3) Client and server tank states now stay in sync after a server-side drain. `saveAdditional` only writes the `sectionInput` / `sectionOutput` keys when the corresponding tank is non-empty (a reasonable size optimisation), but the matching `loadAdditional` was guarded by `if (!fluid.isEmpty())` and so silently kept the previous client-side value whenever the server's `getUpdateTag` round-trip omitted the key. Visible symptom (per F3 debug overlay): server saying `tank_output = empty` while the client still rendered the tank as `1000 mB buildcraftunofficial:oil_heat_1`. The four tank loads now route through a small `loadTank` helper that always calls `set(0, ...)` ‚Äî passing `(EMPTY, 0)` when the saved value was empty ‚Äî so client tanks track every drain. New `testTankClearsWhenLoadedFromEmptySave` regression test covers the round trip. (4) The four bucket / fluid-shard slots are constructed as `new ItemHandlerSimple(4, 1)`, but `ItemHandlerSimple.getCapacityAsLong` was hard-coded to 64 ‚Äî only the `insert(...)` path was capped at 1. The slot widget reads its max-stack via that capacity (through `SlotBase.getMaxStackSize`), so vanilla container code happily dropped a 64-stack of (modded, stackable) buckets into a slot meant to hold one. The auto-fill loop then ran `tryFillContainer` on the whole stack, got back a single filled bucket, and `setStackInSlot`'d it on top of the original ‚Äî silently deleting the rest. Fix: `ItemHandlerSimple` now stores the configured max-stack in a `slotCapacity` field that `getCapacityAsLong` returns, so the slot widget enforces the real limit. As defence-in-depth, `drainSlotIntoTank` and `fillSlotFromTank` bail out when the slot somehow has more than one item, so even a regressed limit can't trigger the data loss again. New `testItemHandlerRespectsConfiguredMaxStackSize` regression covers both the capacity readout and the inserter cap.
- Restore the 1.12.2 Heat Exchanger fluid restrictions. The port lost both `tankInput.setFilter(isHeatant/isCoolant)` and `tankOutput.setCanFill(false)`, so any fluid (lava, water, oil, ...) could be inserted into any tank of the multi-block via a side-touching pipe, a bucket right-click, or a click in the GUI's tank widget ‚Äî making the START accept coolants, the END accept heatants, and both output tanks accept arbitrary external fluids. The START's `tankInput` now overrides `isValid` to accept only fluids registered with `IRefineryRecipeManager.getHeatableRegistry()` (water, oil, fuel, ...) and the END's `tankInput` only fluids in `getCoolableRegistry()` (lava, hot oil, hot fuel, ...). Both output tanks are now instances of a new `OutputTank` subclass whose `isValid` returns false to externals; the craft loop fills them via `insertInternal`, which flips a guard for the duration of one `super.insert` call so recipe results still land. NBT load uses `set()` directly to bypass both the heatant/coolant filter and the output gate (saved fluid is trusted, and the recipe registry may not be ready during early chunk load). New `HeatExchangerTester` covers the OutputTank gate in three game-tests: external insert returns 0, internal insert succeeds for the same fluid, and the internal-insert flag resets after the call so a follow-up external insert is still rejected.
- Wire up the Heat Exchanger GUI. The `heat_exchanger.png` asset has been shipped since the port but no `Container`/`Screen`/`MenuType` ever pointed at it, so right-clicking only ever did bucket transfer. `TileHeatExchange` now implements `MenuProvider`, `BlockHeatExchange` opens the menu after a failed bucket interaction (matching the Distiller pattern), and right-clicking any block in the multi-block walks clockwise to find the START tile via the new `findStart()` helper so the menu always points at the section that owns the recipe state and the `endSection` link. The four 1.12.2 bucket / fluid-shard container slots are wired up as a `(4, 1)` `ItemHandlerSimple` on `TileHeatExchange`: every 5 ticks the START's `processContainerSlots()` drains filled containers in slots 0/1 into END.tankInput / START.tankInput respectively and fills empty containers in slots 2/3 from END.tankOutput / START.tankOutput, with `quickMoveStack` shift-click ordering between the four slots and the player inventory; slot contents persist via NBT and drop on block break. The center processing region between the tanks gets a new wipe-in/wipe-out overlay sourced from `(176,71) 54x71`: as `ExchangeSectionStart.progressState` advances OFF‚ÜíPREPARING‚ÜíRUNNING‚ÜíSTOPPING‚ÜíOFF, the overlay wipes in from the left during PREPARING (visible width grows with `getProgress(partialTicks)`), holds at full 54√ó71 during RUNNING, then clears from the left during STOPPING (the visible region's left edge marches rightward) ‚Äî visualizing the two fluids meeting in the middle and then dispersing. The vanilla title and inventory labels are suppressed because the layout is too cramped to fit either without overlapping the pipe schematic or the slot rows. Reuses the existing `block.buildcraftunofficial.heat_exchange` lang key for the menu title.

- Move the entire visible logic-gate body from chunk-mesh-baked geometry to per-frame BlockEntityRenderer rendering, eliminating the client-side FPS spikes (~6 ms ‚Üí 80‚Äì95 ms) that occurred whenever a gate changed state ‚Äî every gate ON/OFF transition (e.g. a timer-driven gate firing every 10 ticks), every redstone toggle on a driven block (the lamp's `setBlock(UPDATE_ALL)` cascading back through the pipe's `neighborChanged`), and every block placement within the 27-section dirty cube around a pipe network. The chunk-mesh path put `logic.isOn` and the gate's static parts (base material brick, logic icon, modifier dots) into the chunk geometry hash, so each transition invalidated the cached pipe geometry and forced a re-mesh whose per-vertex AO + lighting cost scaled linearly with the gate count nearby. Chunks with N gates re-mesh at the same speed as chunks with 0 gates; gates render at proper world brightness with vanilla face shading by sampling lightmap from `pipePos.relative(plug.side)` and keeping `createFace`'s correct outward normals. Several smaller fixes ride along: `Pipe.updateConnections` only schedules its network update when `connected` actually changes (cuts a feedback loop where the lamp's UPDATE_NEIGHBORS bounced back through the pipe and forced a full BE sync every redstone cycle), `TilePipeHolder.replacePluggable` rebuilds the wire system when an `IWireEmitter` pluggable is added or removed (long-standing bug where placing a gate after wires already existed meant the wires never glowed because the wire system was built without that emitter element), and the four `Block.UPDATE_ALL` ‚Üí `Block.UPDATE_CLIENTS` calls on the pipe-holder render-sync paths drop a duplicate server-side `updateNeighborsAt` cascade.
