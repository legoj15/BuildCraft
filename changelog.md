###### Changes since 26.1.x Beta release 6:
- Move the entire visible logic-gate body from chunk-mesh-baked geometry to per-frame BlockEntityRenderer rendering, eliminating the client-side FPS spikes (~6 ms → 80–95 ms) that occurred whenever a gate changed state — every gate ON/OFF transition (e.g. a timer-driven gate firing every 10 ticks), every redstone toggle on a driven block (the lamp's `setBlock(UPDATE_ALL)` cascading back through the pipe's `neighborChanged`), and every block placement within the 27-section dirty cube around a pipe network. The chunk-mesh path put `logic.isOn` and the gate's static parts (base material brick, logic icon, modifier dots) into the chunk geometry hash, so each transition invalidated the cached pipe geometry and forced a re-mesh whose per-vertex AO + lighting cost scaled linearly with the gate count nearby. The new `PlugGateRenderer` (using the existing `IPlugDynamicRenderer` plumbing the pulsar already had) renders all gate geometry per-frame from the live `PluggableGate.logic.isOn`, `PlugGateBaker.bake` returns an empty quad list, and `KeyPlugGate.equals/hashCode` exclude `active` so the geometry hash is stable across ON/OFF. Chunks with N gates re-mesh at the same speed as chunks with 0 gates; gates render at proper world brightness with vanilla face shading by sampling lightmap from `pipePos.relative(plug.side)` and keeping `createFace`'s correct outward normals. Several smaller fixes ride along: `Pipe.updateConnections` only schedules its network update when `connected` actually changes (cuts a feedback loop where the lamp's UPDATE_NEIGHBORS bounced back through the pipe and forced a full BE sync every redstone cycle), `TilePipeHolder.replacePluggable` rebuilds the wire system when an `IWireEmitter` pluggable is added or removed (long-standing bug where placing a gate after wires already existed meant the wires never glowed because the wire system was built without that emitter element), and the four `Block.UPDATE_ALL` → `Block.UPDATE_CLIENTS` calls on the pipe-holder render-sync paths drop a duplicate server-side `updateNeighborsAt` cascade. Pulsar pulse-state transitions ride a 3-byte custom payload via `IPipeHolder.sendMessage` instead of forcing a full BE sync via `scheduleNetworkUpdate`.
- Bump NeoForge to `26.1.2.30-beta` (was `.17`). Pulls in mod-loading restructure, `ValueInput#keySet` / `FlatLevelSource` / `ComposedModelState` / `TagEmptyCondition` fixes, fail-fast on configs whose defaults don't accept their own validators, and refreshed Sponge Mixin / FancyModLoader / NeoForge Bus dependencies. None of the breaking changes in the range (GLM priority system, `BreakBlockEvent` split, `IBaseRailBlockExtension` hook removal) touch this codebase.
- Restore the six "face the &lt;direction&gt; side" gate actions for iron, wood, and daizuli pipes; the action icons were missing from the gate GUI's action picker and triggering them did nothing. Each pipe filters which directions are offered via its existing `canFaceDirection` rule (iron and wood require an active connection, daizuli accepts all six).
- Restore the 16 "Paint Items &lt;colour&gt;" gate actions on lapis and daizuli pipes. Two regressions: no pipe added them to its gate's possible-actions list (so they never appeared in the action picker), and the sprite holder pointed at `items/paintbrush/<colour>` (legacy 1.12.2 path) instead of the canonical `item/paintbrush/<colour>` so the icons rendered as the magenta missing-texture box even after the actions were re-registered. Triggering one now sets the pipe's paint colour the same way the wrench does.
- Teach `SpriteHolderRegistry.SpriteHolder.getSprite()` to walk multiple texture atlases (blocks → items → GUI) instead of hard-coding `LOCATION_BLOCKS`. Modern Minecraft (1.21+) split the historical single atlas into per-purpose atlas pages; sprites referenced by item models — like the paintbrush colour textures gate "Paint Items" actions reuse for their icons — auto-stitch into `textures/atlas/items.png` and were unreachable from the blocks-atlas-only lookup, rendering as the missing-texture box on the guide-book contents page. The fall-back resolves each PNG against its actual atlas page so each texture lives on exactly one atlas (no double-stitching, no `Duplicate sprite … will be rejected in a future version` warnings) while remaining accessible to all callers; resolution caches on first query so the cost is paid once per holder. Also dropped the redundant `single` declarations of `gui/empty_filtered_buffer_slot` and `gui/nothing_filtered_buffer_slot` from `assets/minecraft/atlases/gui.json` — both sprites are already covered by the `gui/` directory source on the blocks atlas, and the SpriteHolder fall-back finds them there without the duplicate warning.
- Guide book polish:
  - Restore the 1.12.2 search behaviour where the guide book's search box matches every registered item and fluid, not just ones with a BuildCraft markdown page. Vanilla and other-mods' items/fluids now appear in search results (under hidden "Other Items" / "Other Fluids" chapters that only surface during a search), and clicking a non-guide item opens an auto-generated stand-in recipes page just like it did in 1.12.2. Group sub-pages — individual filler patterns, the four Emzuli extraction presets, the per-colour pipe-signal triggers/actions, and the per-direction "Set Pipe Direction" actions — are also indexed so typing e.g. "flatten" or "green extraction preset" once again returns a hit even though those statements stay collapsed under their category entry in the default contents view.
  - Fix the open Guide Book GUI rendering blank or unresponsive content after `/reload` or a datapack sync. Open pages now rebuild themselves against the freshly-loaded registry on the next tick (preserving search text and page position) instead of holding stale references that required closing and reopening the book to recover.
  - Fix the configuration and gameplay guides showing identical content on the first open. The guide registry is now populated synchronously before the guide-book screen is constructed, so book identifiers resolve correctly on every open.
  - Side bookmark tabs that have an expand/collapse arrow no longer slide on hover, so the arrow stays in place and is easier to click.
  - Side bookmark tabs now clamp their hover-extension and wrap their title text onto multiple lines when the screen is too narrow, instead of animating off-screen or overflowing past the viewport.
  - Eliminate the 1-pixel jitter between an animating tab and the book by snapping nine-sliced sprite endpoints to whole pixels.
  - Multi-line clickable chapter entries on the contents page now highlight as a single cohesive block instead of each wrapped line behaving like a separate clickable.
  - Clicking an entry's icon (on contents/index pages and on group/category pages) now opens that entry, matching the affordance the icon-hover already suggested. Hovering either the icon or the text now lights up the whole row and swaps the icon to its hovered drawable, so icon and text read as a single click target.
  - Collapse the ~19 individual filler-pattern entries that previously cluttered the Actions chapter alphabetically into a single "Filler Patterns" entry (stairs icon). The page opens with a short description of what filler patterns do and lists every pattern in the same order the filler's GUI uses; each pattern remains clickable to navigate to its own page.
  - Same treatment for Emzuli extraction presets: the four colour-keyed preset actions now collapse into one "Emzuli Extraction Presets" entry under Actions (red preset icon), with a description page that lists the presets in slot order. The Emzuli pipe's own entry now links to that page in the Filtering section. Removed three manual links to the redstone engine, pipe power adaptor, and pipe pulsar from the Emzuli pipe page since the auto-emitted "Linked To" chapter already covers them. Markdown `<link to="..."/>` tags can now resolve to category pages, not just registered guide entries.
  - Same treatment again for Pipe Signals: the 16 colours × (1 action + 2 triggers) = 48 statements that previously sprawled across the Triggers and Actions chapters (under Basic, Pipe Transport, Pluggables, …) collapse into one "Pipe Signals" entry, filed under both Triggers AND Actions so it shows up wherever a reader would have looked for the per-colour leaves. The page reuses the 1.12.2 "Black Pipe Signal" wording, covers both action and trigger semantics in the same body, and lists every variant in colour-major order so a Black/Red/… reader still finds them. The Pipe Wire entry gains a link to the new category page. Category registration helpers now accept multiple chapter tags, so other dual-chapter categories can use the same one-page pattern.
  - Same treatment again for the six "Face the &lt;direction&gt; side" gate actions: collapse them into a single "Set pipe direction" entry under Actions, using a vanilla compass as the icon. The page opens with a short description of which kinds of pipes the actions apply to and lists all six directions in Minecraft's canonical order; each per-direction page stays as-is and remains clickable from the listing. The compass icon is frozen on `compass_16` (the needle-straight-up frame, threshold-0.0 entry of vanilla's `items/compass.json` range_dispatch) by blitting that sprite directly via `SpriteHolderRegistry.SpriteHolder` instead of rendering an `ItemStack(Items.COMPASS)` — without a level/entity context the dispatch falls back to a time-based wobble, so a plain item icon would spin on the contents page. Resolution goes through the multi-atlas walker because modern Minecraft (1.21+) stitches item textures onto a separate `textures/atlas/items.png` rather than the historical unified blocks atlas, so a hard-coded `LOCATION_BLOCKS` lookup returns the missing-texture sprite (same regression class the paintbrush sprites hit). The fixed-frame blit still picks up resource-pack retextures because packs replace the per-frame `compass_XX.png` textures.
  - Wire the guide into the client resource-manager reload pipeline so F3+T regenerates guide content against edits to the resource pack. The reload-listener method existed but was never registered with NeoForge's `AddClientReloadListenersEvent`, so editing a guide `.md` file and pressing F3+T appeared to be a no-op. The listener no-ops on the very first reload (which fires during `Minecraft` construction before BC item Holders have their components bound — that path threw `Components not bound yet` and crashed the loading screen to black); the first real load still happens via `ensureLoaded()` on guide-book open, and subsequent F3+T reloads proceed normally. (`/reload` doesn't apply — it's for server-side datapacks, while guide content lives in resource packs.)
  - Restore smooth interpolation on the bookmark-tab hover animation (and the cover-flip when the book opens) by using Minecraft 1.21+'s partial-tick fraction instead of the per-frame delta that was being incorrectly substituted.
  - Stop double-defining the page title on entries (e.g. Auto Workbench) that previously had a redundant `<chapter name="tile.X.name"/>` at the top of their markdown. The auto-prepended page-title chapter now contributes only a side-tab + jump anchor, never an inline body heading.
  - Render the page title centered over each visible page rather than over the spine — readable on every spread, and skipped on the back-of-book half-page when an entry ends on an odd page.
  - Restore the "Linked From" / "Linked To" chapters on entry pages. The combustion (iron) engine page now lists every registered fuel, the heat exchange lists its coolants and recipe fluids, the distiller lists its inputs and outputs, and the four conceptual groups from 1.12.2 (pipe power, full power, laser power, area markers) are restored. New `fluid_stack` page-entry type lets fluid endpoints render as proper fluid icons. Group entries are now clickable to navigate to the target page, and `<group direction="from"/>` lets markdown override the default direction.
  - New `<link inline="..."/>` markdown directive renders a clickable hyperlink mid-paragraph, with no icon column and no row break. Authors can now drop a link inside a sentence (e.g. `See the <link inline="buildcraftunofficial:engine/stirling"/> for one option.`) instead of forcing the existing `<link to>` form, which inserts a TOC-style icon row. The shorthand puts the target directly in the `inline` attribute; the longer `<link inline="true" to="..."/>` form also works and is required when adding `type="..."` (typed links). Composes with surrounding inline formatting (`<bold>`, colours).
  - Fix a latent paragraph word-wrap bug where, when a segment with non-default formatting couldn't fit any of its content on the current row, the wrap built the next-row remainder by scanning from `segments[1]` instead of `segments[segmentIndex]` — duplicating already-emitted segments and dropping trailing ones. Manifested as e.g. mid-paragraph words appearing twice and the tail of a paragraph silently disappearing. Stayed latent because plain `<bold>`/colour spans are short enough to fit, and block-level `<link to>` is on its own line; the new mid-paragraph `<link inline>` finally produced a paragraph long enough with enough style-bearing segments to trigger the path reliably.
