###### Changes since 26.1 Beta release 5:
- Fix: Repaired the three game tests that were failing after the previous changelog entry restored visibility. (1) `gui_test_dynamo_upgrade` — its spec referenced `buildcraftunofficial:iron_gear`, but the item is registered as `gear_iron`, so the `.orElseThrow()` on the item-registry lookup threw `NoSuchElementException: No value present` before the test even opened the menu; spec now uses the correct id. Its shift-click `slot` was also pointing at an empty main-inventory slot (36) instead of the hotbar slot (31) where `set_hotbar` had placed the gear, and `TileDynamoMJ` didn't implement `MenuProvider`, so the test's `instanceof MenuProvider` check at menu-open time silently no-op'd — fixed both. (2) `gui_test_filler_ui` had the same 36-vs-54 slot-index bug, plus the final assertion checked for pattern tag `buildcraftunofficial:fill` when `Pattern`'s tag format is `buildcraft:<name>` (and the spec was sending `buildcraft:cylinder`, which isn't a real pattern, so `FillerType.readFromBuffer` fell back to `PATTERN_NONE`). (3) `oil_bobbing_physics` — the oil buoyancy logic was living in `BCCustomFluidType.move(...)`, but `FluidType.move` is a dead override point in 1.21.x (its vanilla default just returns `false`; nothing in patched `Entity`/`LivingEntity` calls it), so our pig-float code never ran and pigs stayed glued to the floor. Moved buoyancy onto a new `BCBuoyantLiquidBlock extends LiquidBlock` via the real `entityInside` hook; all BC fluids now route through it.
- Fix: Restored 15 previously-invisible game tests to the `runGameTestServer` run. In 1.21.11+, a test needs both a `TEST_FUNCTION` (Java) registration and a `test_instance` datapack JSON; the server enumerates the datapack registry, so functions without a matching JSON were silently skipped (only 19 of 33 were running). Added `test_instance` JSONs for the missing 15 (fluid physics, filtered-buffer drops, dynamo/FE-engine upgrade + GUI, filler filter, slot max-stack, builder/architect drops, GUI spec tests).
- Fix: Fixed a server crash in `BlueprintBuilder.cancelPlaceTask` when a fluid place task was cancelled after the fluid-placement deadlock fix enabled them to actually queue. The extract path's "pack fluid info into the resulting bucket item's CustomData" step had been TODO-stubbed to write an empty `BuilderFluidStack` tag, and the corresponding cancel-path `map(stack -> return FluidStack.EMPTY)` then fed `FluidResource.EMPTY` into `tankManager.insert`, tripping NeoForge's `checkNonEmptyNonNegative` precondition with `IllegalArgumentException: Expected resource to be non-empty: minecraft:empty [0]`. Now the extract path actually serialises the fluid id + amount into the bucket's CustomData, and cancel reads it back out to reconstitute the FluidStack for the tank refund — with a defensive filter so any empty/unresolvable fluid (old save data, unregistered fluid) is skipped instead of crashing.
- Fix: Builder now actually places source fluid blocks (water, lava, oil, fuel, …) that appear in a blueprint. Each `SchematicBlockFluid` listed its four horizontal neighbours + the block below in `getRequiredBlockOffsets()`, so `BlueprintBuilder.isReadyToPlace` made every planned source fluid wait for every adjacent planned source fluid to already be `CHECK_RESULT_CORRECT` — for any blueprint where two sources sat next to each other (i.e. every pool) they deadlocked each other and nothing ever placed. `isReadyToPlace` now skips the fluid-to-fluid constraint specifically: source blocks don't need each other placed first (the flowing overspill one creates in the other's cell gets overwritten the moment that cell's source places). Fluid-vs-solid constraints (wait-for-floor, wait-for-walls) are preserved.
- Fix: Corrected block loot-table data path from `loot_table/block/` (singular) to `loot_table/blocks/` (plural) so vanilla's datapack loader actually finds them. Without this the Builder and Architect Table had the loot-table JSONs sitting in an unread directory, so breaking either block in survival with a pickaxe dropped the tile contents but not the block item itself. Same typo was silently shipping for the `filtered_buffer_drops` game test (which was hidden until the `test_instance` gate was fixed) — switched that tester to the survival-player + wooden-pickaxe mining flow so it actually exercises the drop path. Also bypassed the `(ServerPlayer) helper.makeMockPlayer(...)` ClassCastException in `GuiTester.open_menu` by instantiating the block's menu directly from its MenuProvider, and implemented the long-commented-out `click_slot` action via `AbstractContainerMenu.quickMoveStack` for the shift-click case the existing specs use.
- Fix: Breaking the Builder AND the Architect Table now drops the block itself (via new loot tables) plus all of their contents. Builder drops: snapshot slot item, every non-empty resource slot (27 total), and each of the 4 fluid tanks as fluid-shard items. Architect drops: snapshot-in and snapshot-out slots. Matches the 1.12.2 behaviour where `BlockBCTile_Neptune.breakBlock` delegated to the tile's own `onRemove`. Mining restriction now also matches 1.12.2: a pickaxe (any tier, wooden+) is required to yield drops, matching the 1.12.2 `Material.IRON` behaviour. Added both blocks to the `minecraft:mineable/pickaxe` tag and set `.requiresCorrectToolForDrops()` on their block properties. Added game tests `buildcraftunofficial:builder_drops_contents_and_self` and `buildcraftunofficial:architect_drops_contents_and_self` that simulate a survival-mode player with a wooden pickaxe running the full `playerWillDestroy → removeBlock → playerDestroy` flow and assert every expected item entity lands in the world.
- Restored Builder block functionality: the block now loads blueprints/templates from its snapshot slot, extracts matching items from its 27-slot resource inventory, pulls fluids from its 4 tanks (8 buckets each, via `FluidStacksResourceHandler`), and places blocks in-world using the already-ported `SnapshotBuilder` / `BlueprintBuilder` / `TemplateBuilder` pipeline. Owner `GameProfile` is now stored on place so `TemplateAPI`'s fake-player path works. The GUI gained the snapshot input slot, 27 resource slots, 24 read-only display slots (synced from the server over a container message so players see which blocks are still needed), 4 fluid tank widgets with tooltips, and left-to-break / left-to-place progress readouts. Also fixed the "blueprint door always open" visual defect by porting the 1.12.2 `SNAPSHOT_TYPE` blockstate property plus the three door submodels (`slot_empty` / `slot_blueprint` / `slot_template`) and composing them into the block's multipart blockstate.
- Architect Table: scan progress bar in the GUI now actually fills as blocks are captured (previously sampled an empty region of the texture atlas). Blocks currently being scanned are also briefly highlighted in-world with a translucent green "digitizing" cube that fades over ~2.5s, porting the 1.12.2 scan-feedback effect. Positions stream server→client in batches keyed to the architect's chunk, so the effect only transmits to nearby players.
- Architect Table GUI now shows a live 3D preview of the current scan-area contents before a blueprint is built. The preview refreshes while the GUI is open so block edits inside the marker volume surface within ~2 seconds; once a real blueprint exists in the output slot, that takes over as before.
- Blueprint tooltip preview now renders in a dedicated 100×100 panel below the item's main tooltip (matching the 1.12.2 separate-panel behavior), replacing the inline tooltip-image slot that left dead space alongside wider tooltip text. The panel uses the vanilla tooltip background so it picks up resource-pack theming; the 3D preview and rotation are unchanged.
- Blueprint/Template tooltip lines below the item name (blueprint name, hash, date, owner UUID) now render in gray, restoring the 1.12.2 look; modern Minecraft stopped auto-graying custom tooltip lines, which made them read too loud against the item title.
- Blueprint preview rendering now evaluates diffuse lighting in model space (per-frame UBO update compensating for the yaw/pitch transform), eliminating the brightness pulse on side faces as the preview rotated. Top faces render at full brightness, sides at ~60%, bottom at ~20% — close to the 1.12.2 look, with no orientation-dependent flicker.
- Blueprint preview now renders fluid cells (water, lava, oil, fuel, etc.) instead of silently skipping them. Source blocks draw as full textured cubes; flowing cells draw at their actual flow height, so captured cascades and pour streams read correctly in the preview the way they did in 1.12.2. Same-type fluid neighbours get shared-face culling so pool interiors show only the outer shell (no translucent-blend artifacts on water). Water uses the entity-translucent sheet; every other fluid uses entity-cutout so tinted BC fluids don't inherit water's semi-transparency.
- Fix: Architect Table now persists the contents of its input and output slots across chunk unloads / world saves. In 1.12.2 these were auto-persistent `ItemHandlerSimple` fields; the 26.1 port replaced them with raw `ItemStack` fields and forgot to add save/load hooks, so any clean blueprint left in the input or any used blueprint left in the output silently vanished on exit. The slot contents now round-trip via the tile's NBT like every other inventory.
- Fix: Completed the legacy `textures/blocks/` (plural, 1.12.2 convention) to `textures/block/` (singular, modern Minecraft) migration. Deleted 129 duplicate textures, moved 98 orphaned-legacy textures to the modern path, retargeted the `buildcraftunofficial:atlases/blocks.json` source to scan `block/` (removing the redundant `block/plugs` entry), and rewrote all remaining `buildcraftunofficial:blocks/` references across model/blockstate JSONs and Java (including `EnumEngineType` and two disabled renderers) to `buildcraftunofficial:block/`. Resolves missing-texture rendering for sprites such as the Architect Table's digitizing cube (`scan.png`) that the modern model references could not see.
- Fix: Fixed a NullPointerException crash when placing a blueprint in the architect table with no owner (e.g., in creative with no explicit owner set). The snapshot header serialization now uses a zero UUID as a fallback.
- Fix: Addressed an issue causing the BuildCraft Filler to incorrectly render the flying block animation when unpowered.
